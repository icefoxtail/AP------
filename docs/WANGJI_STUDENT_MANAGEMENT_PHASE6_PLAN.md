# 6차 세부계획서 — 기존 학생관리 보조화 (PHASE6_PLAN)

> 이 문서는 6차 CODEX_TASK 지시서의 원본이다.
> 상위 문서: `WANGJI_STUDENT_MANAGEMENT_PHASE_ROADMAP.md` / 선행: 2~5차 완료 + 5차 운영 안정 확인

## 0. 목표

```text
왕지 공통 학생관리가 주 입력 화면이 되고,
기존 AP/EIE 학생관리는 보조/레거시 입력 화면으로 전환한다.
(기존 화면 삭제·대체 아님. 진입점과 운영 관행의 전환이다.)
```

## 1. 사전 결정

| 항목 | 후보 | 결정 |
|---|---|---|
| 메뉴 노출 범위 | 1단계 원장(admin)만 → 2단계 교사 | ☐ |
| 교사 권한 모델 | 교사는 자기 담당 학생만 / 자기 학원(AP 또는 EIE) 학생만 / 전체 | ☐ |
| 메뉴 명칭 | "통합 학생관리" (확정 필요) | ☐ |
| 기존 화면 입력 제한 여부 | 제한 없음(병행) vs 안내 배너만 | ☐ |

## 2. 작업 항목

### 2.1 메뉴 정식 노출 (이때 처음 기존 메뉴에 항목 추가)
- AP(`apmath/index.html` 등) 원장 메뉴에 "통합 학생관리" 항목 1개 추가 → 독립 페이지로 이동.
- EIE 측 노출 여부는 사용자 결정 후. 추가 시 메뉴 항목 1개만.
- 기존 메뉴 항목의 위치/순서/동작 변경 0.

### 2.2 기존 상세 → 통합 화면 버튼
- AP 학생상세에 "통합 학생관리에서 보기" 버튼 추가 (`source_app=AP&source_student_id=` deeplink).
- EIE 학생상세에 동일 버튼 추가.
- **버튼 추가만.** 기존 버튼/로직 제거·변경 0. 해당 학생의 link가 없으면 통합 화면에서 "연결 후보 생성" 흐름으로 안내.

### 2.3 교사 권한 적용
- 사전 결정된 권한 모델을 공통 Worker에서 강제 (프론트 숨김만으로 처리 금지).
- 권한 밖 학생은 목록/검색/상세에서 제외.

### 2.4 운영 전환
- 통합 화면 상단의 "안전 모드" 안내를 정식 운영 문구로 조정.
- 운영 가이드 문서: 신규 등록/상담/메모는 통합 화면 우선, 기존 화면은 보조.
- 데이터 정합성 모니터링: 주간 점검 체크리스트(링크 무결성: source_student_id가 원본에 실존하는지 read로 검사하는 점검 화면 또는 스크립트).

### 2.5 write-through 점진 확대 (선택)
- 5차 화이트리스트 확대(예: enrollment 요청 반영)는 **각 확대마다 5차 수준의 사전 결정+리허설+검수 반복.** 이 차수에 자동 포함되지 않는다.

## 3. 수정 허용 파일

```text
apmath/index.html (메뉴 1항목 + 상세 버튼 진입부 최소)
apmath/js/student.js (상세 버튼 1개 추가만)
eie/index.html / eie/js/views/eie-students.js (버튼 1개 추가만, 사용자 승인 시)
apmath/wangji-student-management.{html,js,css}
workers/wangji-common-worker/* (권한 모델)
```

각 기존 파일은 "추가만" 원칙 — diff에서 기존 라인 삭제/변경이 보이면 실패.

## 4. 금지 (영구 포함)

- 기존 AP/EIE 학생관리 화면 삭제/대체 강행
- 기존 메뉴 항목 제거/재배치
- 원본 DB 폐기, 단일 DB 강제 통합 (별도 마이그레이션 설계 라운드로만)
- 권한을 프론트에서만 처리

## 5. 검증 / 완료 기준

```text
기존 메뉴/버튼 전부 기존과 동일 동작 (수동 회귀)
새 메뉴/버튼: 통합 화면 정상 진입 + 해당 학생 자동 선택
권한: 교사 계정으로 권한 밖 학생 접근 시 서버 403
git diff: 기존 파일은 추가 라인만
정합성 점검 절차 1회 실행 기록
```

# ARCHIVE_EXAM_TARGET_SELECTION_NEXT_PLAN

## 1. 현재 상태

`archive/index.html`의 "출제 대상" 흐름은 진입 시 모달 하나(`targetModalOverlay`)에서 **반별 / 학년별** 중 하나를 고르게 하고, 이후 완전히 분리된 두 모달로 갈라진다.

- 반별: `targetModalOverlay` → `classModalOverlay` 진입 → 선생님 `<select>` → 반 `<select>`(단일 선택) → `confirmClassSelectionForIndex()` → `registerIndexClassExamAssignment()` 1회 호출(`target_scope: 'class'`)
- 학년별: `targetModalOverlay` → `gradeModalOverlay` 진입 → 학년 `<select>`(단일 선택) → `confirmGradeSelectionForIndex()` → 해당 학년의 모든 반을 순회하며 `registerIndexClassExamAssignment()`를 N회 호출(`target_scope: 'grade'`, 공유 `assignment_batch_id`)

두 흐름은 서로 다른 함수/모달로 완전히 중복 구현되어 있고, "반 몇 개만" 또는 "반 안에서 학생 일부만" 고르는 중간 단계가 없다. 배정 저장은 항상 반 단위 1 row다(`class_exam_assignments`: `class_id NOT NULL`, `student_id` 컬럼 없음). 학생 단위로 존재하는 유일한 개념은 사후 제외(opt-out)뿐이다: `class_exam_assignment_exclusions`(`assignment_id`, `student_id`) 테이블과 `exams.js`의 `exclude-student` 엔드포인트(1명 단위).

학생이 보는 배정 목록은 `student-portal.js`가 `class_exam_assignments JOIN class_students WHERE cs.student_id = ?`(본인 반 소속 여부)만으로 판단한다. 교사가 어떤 QR/인쇄 링크를 열었는지와 무관하게 DB 조인만으로 학생별 노출이 결정되므로, 반을 여러 개 동시에 배정해도 노출 로직 자체에는 위험이 없다(확인됨).

추가로 확인한 활용 가능 데이터: `attendance`(student_id, status, date) 테이블이 존재해 "결석 학생 제외" 같은 프리셋을 스키마 변경 없이 구현할 수 있고, `exam_sessions`(student_id, exam_title, exam_date)로 "같은 시험을 이미 응시한 학생"을 걸러낼 수 있다(재시험/보강 대상만 뽑기). 반면 수납/미납 상태는 별도 billing-accounting-foundation 모듈에 있고 학생-결제 연결이 이번 범위 데이터만으로는 신뢰성 있게 조인되지 않아, "미납생 제외" 프리셋은 이번 계획에서 **제외**한다(6절 참고).

반 소속 학생 명단을 `archive/index.html`의 인증 컨텍스트에서 조회할 API가 현재 없다(`/qr-classes`는 반 목록만 반환). 다건 배정/다건 제외 시 **부분 실패 처리 UX도 현재 없음** — `confirmGradeSelectionForIndex()`의 `for...await` 루프는 중간에 하나라도 실패하면 전체를 `catch`로 던지고 일반 오류 문구만 보여주는데, 이미 성공한 앞쪽 반들의 배정은 롤백되지 않아 "어디까지 성공했는지 교사가 알 수 없는" 상태가 될 수 있다.

## 2. 최종 목표

"반별 vs 학년별" 분기 선택을 없애고, 학년 → 반 → 학생으로 내려가는 **하나의 계층형 화면**으로 통합한다. 모달 크기 제약은 두지 않고(선생님이 넓은 화면에서 편하게 훑어볼 수 있도록 큰 패널/시트로), 반과 학생을 한 화면에서 동시에 비교하며 고를 수 있게 한다.

1. 학년만 고르고 멈추면 → 학년 전체 출제 (기존 학년별과 동일 결과)
2. 반 체크리스트에서 일부만 체크하면 → 체크한 반들에만 출제 (신규: 반 다중 선택)
3. 특정 반의 학생 목록에서 일부만 체크하면 → 그 반은 체크한 학생에게만 출제, 나머지는 자동 제외 (신규)

스키마 변경은 하지 않는다. 배정은 여전히 반 단위 row + 필요 시 제외 row 조합으로 표현한다.

## 3. UI/UX 설계 (선생님 친화성 최우선)

### 3.1 설계 원칙

- **분기 없는 단일 흐름**: "반별/학년별" 버튼(`targetModalOverlay`)을 없애고 하나의 "출제 대상 선택" 화면만 연다.
- **모달 크기는 자유롭게, 대신 클릭 수를 줄인다**: 크기 제약이 없으므로 아코디언으로 숨기지 않고, 체크된 반의 학생 목록을 **항상 펼친 상태로 나란히** 보여준다. 여러 반을 오가며 학생을 비교하기 위해 접었다 펼쳤다 할 필요가 없어야 한다.
- **기본값 = 가장 흔한 동작**: 반/학생 체크박스는 기본 전체 체크. 아무것도 건드리지 않고 확인을 누르면 지금의 "학년별 전체"와 동일하게 동작한다(회귀 없음, 학습 비용 없음).
- **숫자보다 이름**: "8/12명 선택" 같은 카운트만으로는 어떤 학생이 빠졌는지 바로 알 수 없다. 제외된 학생은 항상 **이름 칩(chip)**으로 보여주고, 칩을 클릭하면 즉시 되돌릴 수 있게 한다.
- **자주 하는 작업은 프리셋 버튼으로**: 매번 한 명씩 체크 해제하는 대신, "결석생 자동 제외", "이 시험 이미 응시한 학생 제외(재시험용)" 같은 원클릭 프리셋을 제공한다.
- **최종 확인은 이름 목록으로**: 실제로 배정이 걸리는 행동이므로, 확인 버튼 이전에 "누구에게 나가는지"를 이름 단위로 다시 보여주는 리뷰 단계를 둔다(오배정 방지).
- **실패를 숨기지 않는다**: 다건 등록/제외 중 일부가 실패해도 나머지는 계속 진행하고, 끝난 뒤 반별 성공/실패를 표시하며 실패한 반만 재시도할 수 있게 한다.

### 3.3 인터랙션 세부

- **좌우 마스터-디테일**: 왼쪽은 반 체크리스트(스크롤 짧음), 오른쪽은 체크된 반들의 학생을 반별 섹션으로 **모두 펼쳐서** 보여준다. 모달 폭 제약이 없으므로 두 패널을 나란히 둬서 "반을 눌러야 학생이 보이는" 클릭 단계를 없앤다.
- **반 체크박스 3-상태**: 완전 체크(반 전체) / 완전 해제(제외) / indeterminate(부분 선택). indeterminate는 네이티브 `input.indeterminate = true` + 배지 텍스트("8/12명")로 표시.
- **글로벌 학생 검색**: 상단 검색창에 이름을 치면 학년/반 구분 없이 현재 화면에 로드된 모든 학생 중 일치하는 학생으로 스크롤/하이라이트하고 바로 체크 토글 가능(반을 먼저 못 찾아도 학생 이름만 기억하면 됨).
- **제외 학생 칩**: 반 섹션마다 "제외됨: [이름 ✕] ..." 칩 목록을 보여주고, 칩의 ✕를 누르면 즉시 그 학생을 다시 포함시킨다(체크박스를 다시 찾아 누를 필요 없음).
- **프리셋 버튼(반 섹션마다)**:
  - `결석생 제외`: 해당 반 학생 중 최근 결석 기록(`attendance.status`)이 있는 학생을 자동으로 체크 해제하고 토스트로 "3명 자동 제외됨 [되돌리기]" 표시
  - `이 시험 미응시자만(재출제용)`: 같은 `exam_title`로 이미 `exam_sessions`가 있는 학생을 자동 제외해, 결석/재시험 대상만 남긴다
  - 두 프리셋 모두 결과를 즉시 실행 취소할 수 있는 토스트를 동반한다(자동화가 실수를 만들 수 있으므로)
- **shift-클릭 범위 선택**: 학생이 많은 반에서 첫 학생 클릭 후 shift+클릭으로 범위 전체를 토글.
- **최종 확인 화면(리뷰 단계)**: 체크박스 화면과 실제 API 호출 사이에 반드시 "이름 목록" 리뷰 단계를 둔다. 되돌리기(뒤로)가 항상 가능하다.
- **진행 상황 + 부분 재시도**: 반 단위로 성공/실패 아이콘을 실시간 표시. 실패한 반만 재시도(이미 성공한 반은 다시 호출하지 않음).
- **로딩/빈 상태/오류**: 로그인 유도 블록, "등록된 반이 없습니다", 네트워크 오류 안내는 기존 패턴 재사용. 특정 반의 roster 조회 실패는 그 반 섹션 안에만 인라인 오류로 표시하고 나머지 반 선택은 계속 가능해야 한다.
- **모바일**: 좌우 패널은 넓은 화면 기준이며, 좁은 화면에서는 상단 탭("반 선택" / "학생 선택")으로 전환하는 두 화면 구성으로 대체한다(수직 스택보다 tap 전환이 손가락 스크롤 피로가 적음).

## 4. 절대 금지/보류

- `ARCHIVE_OMR_DOMAIN.md`의 기존 정책 유지: 학생 시험지 직접 열기 금지, 일반 OMR 제출 완료 후 수정 금지, archive 원본 path 학생 노출 금지
- `class_exam_assignments`/`exam_sessions`에 `student_id`/`grade_id` 컬럼을 추가하는 스키마 변경 금지 — 기존 exclusion 테이블 재사용으로 대체한다
- 반 인원이 많을 때 제외 처리를 학생 수만큼 무제한 병렬(fire-and-forget) 호출 금지 — 순차/청크 처리 및 실패 항목만 재시도하는 UX 필수
- 다건 등록/제외 중 일부 실패를 조용히 삼키거나 롤백 없이 모호하게 실패 처리하는 방식 금지 — 3.2의 결과 패널(성공/실패 표시 + 부분 재시도) 없이는 배포하지 않는다
- "결석생 제외"/"미응시자만" 같은 자동 프리셋은 반드시 실행 취소(되돌리기) 토스트를 동반해야 한다 — 자동 제외가 조용히 확정되어 특정 학생이 이유 없이 시험을 못 받는 사고 방지
- 수납/미납 데이터를 이번 계획에서 연동하지 않는다(billing 모듈과의 신뢰성 있는 조인 미검증 상태) — 관련 프리셋 UI 노출 금지
- 기존 "반별"/"학년별" 흐름을 검증 없이 한 번에 제거해 진행 중인 출제 워크플로우를 깨뜨리지 않는다 — 신규 통합 화면을 완성/검증 후 기존 두 모달과 `targetModalOverlay`를 제거하는 단계적 교체

## 5. Phase 구조

### Phase 1: 서버 — 조회 API 확장

- `check-omr.js`의 `handleQrClasses` 패턴을 확장하거나 신규 핸들러로 `GET /qr-classes/:id/students` 추가. 응답은 재원(`status='재원'`) 학생만, 최소 필드(`id`, `name`, `school_name`)
- 프리셋 계산에 필요한 부가 조회 추가: 같은 응답 또는 별도 엔드포인트로 (a) 최근 N일 내 결석 여부(`attendance` 조인), (b) 지정한 `exam_title`에 대한 기존 `exam_sessions` 존재 여부를 함께 내려줘서 프론트가 즉시 프리셋 버튼을 계산할 수 있게 한다
- N명 제외를 한 번에 처리하는 배치 제외 엔드포인트 신설: `student_ids` 배열을 받는 `exclude-students` bulk 엔드포인트, 부분 실패 시 개별 student_id별 성공/실패를 응답에 포함(재시도 UX에 필요)

### Phase 2: 프론트 — 마스터-디테일 통합 화면 (`archive/index.html`)

- `targetModalOverlay`(반별/학년별 선택 버튼) 제거, 시험지 카드 클릭 시 바로 신규 대형 패널(`assignTargetPanel`) 오픈
- 3.2 와이어프레임대로 좌(반)-우(학생) 마스터-디테일 레이아웃, 학년 탭, 글로벌 학생 검색 구현
- 상태 모델: `{ grade, classes: { [classId]: { checked: true|false|'partial', studentIds: Set|null } } }` — `studentIds`가 `null`이면 반 전체, `Set`이면 해당 학생만
- 제외 학생 칩 + 되돌리기, 프리셋 버튼(결석생 제외/미응시자만) + 실행취소 토스트 구현
- "최종 확인" 리뷰 화면(이름 목록) 구현 — 체크박스 화면과 실제 API 호출 사이의 필수 단계
- 확인 시 상태를 순회하며: 체크된(전체 또는 partial) 반마다 `registerIndexClassExamAssignment()` 호출 → partial인 반만 미체크 학생 id 목록으로 Phase 1의 `exclude-students` 벌크 호출
- 진행 상황/결과 패널 구현: 반 단위로 성공/실패 상태를 즉시 반영, 실패한 반만 재시도
- `goEngine()` 인쇄/QR 오픈 정책 확정: 다중 반 선택 시 QR 인쇄를 반별로 각각 열지, 대표 1건만 열지는 실제 교사 인쇄 워크플로우 확인 후 결정. 부분 선택된 반(일부 학생 제외)은 인쇄 시 "이 인쇄물은 반 전체가 아니라 8명 대상입니다" 같은 안내 문구를 표시해 착각을 막는다
- 모바일: 좌우 패널을 상단 탭 전환("반 선택"/"학생 선택") 구조로 대체

### Phase 3: 배정/제외 로직 통합 검증

## 8. 검증 기준

- 학년만 선택하고 아무것도 건드리지 않으면 기존 "학년별 전체" 배정과 동일한 결과가 나온다(회귀 없음)
- 반 체크리스트에서 일부 반만 체크하면 체크한 반 수만큼만 `class_exam_assignments` row가 생성된다
- 반 내 학생 부분 선택 시, 선택 안 한 학생 수만큼 제외 row가 생성되고 해당 학생은 student-portal에서 해당 시험이 보이지 않는다
- 반 전체 선택(기존과 동일 동작)일 때는 제외 API가 호출되지 않는다
- 결석생 제외/미응시자만 프리셋이 실제 결석/응시 기록과 정확히 일치하는 학생만 제외한다
- 다건 처리 중 일부 실패 시 성공/실패가 UI에 정확히 구분되어 표시되고, 실패 항목만 재시도로 복구된다
- 최종 확인 화면의 이름 목록이 실제 API 호출 대상과 항상 일치한다
- `class-exam-assignments/board` 조회 시 다중 반/부분 선택 배정이 정상적으로 표시된다

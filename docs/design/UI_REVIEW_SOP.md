# UI 검수 SOP

이 문서는 Gemini, Claude, ChatGPT, Codex, 사람 작업자가 APMS UI 패치를 검수할 때 사용하는 현미경 검수 요청서 기준이다.

목표는 칭찬이 아니라 회귀와 누락을 찾는 것이다.

---

## 1. 검수 기본 자세

검수자는 PASS를 주기 위한 사람이 아니다.

검수자는 이번 패치가 다음 문제를 만들 수 있다고 가정한다.

- 기존 화면이 깨짐
- 카드 shell이 사라짐
- 카드 안 카드가 생김
- 글자 두께/크기 통일이 깨짐
- 전역 CSS가 오염됨
- 기존 기능/문구/버튼명이 바뀜
- 클릭 이벤트가 충돌함
- 저장/병합 로직이 깨짐

근거 없는 PASS 금지.
미검수 항목이 있으면 PASS 금지.
요약만 보고 PASS 금지.

---

## 2. 필수 확인 순서

1. `00_manifest.md` 확인
2. `01_changed_snippets.md` 확인
3. `full_files`의 실제 변경 파일 전체 확인
4. 관련 CSS 전체 확인
5. 관련 JS 함수 전체 확인
6. node --check 실패 가능성 확인
7. 브라우저 화면 캡처 기준 확인 항목 작성

`00_manifest.md`와 `01_changed_snippets.md`만 보고 PASS하면 검수 실패다.

---

## 3. UI 패치 검수 필수 항목

### 3.1 기능 보존

- 기존 버튼명 보존
- 기존 화면명 보존
- 기존 문구 보존
- 기존 클릭 흐름 보존
- 기존 권한 분기 보존
- 기존 저장/삭제/수정 흐름 보존

### 3.2 CSS 안전성

- 전역 selector 수정 여부
- `button`, `.card`, `body`, `*` 수정 여부
- `!important` 남발 여부
- 기존 selector 삭제 여부
- 새 class와 실제 JS 마크업 연결 여부
- inline style 신규 추가 여부

### 3.3 타이포그래피

- font-weight 700 이상 신규 사용 여부
- 숫자만 bold 처리 여부
- 상태값만 과도하게 강조 여부
- 같은 row 안 글자 크기/두께 혼합 여부
- 버튼 글자 두께 통일 여부

### 3.4 카드/row 구조

- 큰 정보 묶음 카드 shell 유지 여부
- 카드 안 카드 발생 여부
- 반복 항목이 line row로 정리됐는지
- 클릭 가능한 row에 chevron이 있는지
- hover 시 layout shift가 없는지

### 3.5 런타임 위험

- 중복 함수 선언
- 닫히지 않은 template literal
- onclick 문자열 escape 문제
- `event` 미정의 가능성
- `state.db` 빈 배열/undefined 접근
- 데이터가 없을 때 화면 크래시 가능성

## 4. 대시보드 검수 전용 항목

대시보드 패치에서는 반드시 확인한다.

- 오늘일지 outer card 유지
- 오늘일지 내부 수/목 row는 카드가 아닌 line row
- 원장 선생님 카드에서 최근 등록 chip 제거 유지
- 원장 선생님 카드에서 일지 확인 버튼 제거 유지
- 담당반 보기 유지
- 수/목 외 요일 렌더링 없음
- 제출완료/미작성 색상 강조 없음
- 상담 병합 로직 유지
- 3단 자동 판단 문구가 다시 살아나지 않음
- 오늘일지 중등부 제한 유지
- 집중케어 50점 하한선 유지
- 성적 기반 집중케어 로직 재유입 없음

## 5. 출력 형식

검수 결과는 반드시 아래 형식으로 출력한다.

```text
1. 최종 판정
PASS / FAIL / CONDITIONAL PASS 중 하나

2. 실제 확인한 파일과 근거
- 파일명:
- 확인한 함수 또는 selector:
- 실제 확인 근거:

3. PASS 항목
- 항목별로 짧게

4. 발견한 오류/위험
- 파일명:
- 함수명 또는 selector:
- 문제 코드 또는 문제 구조:
- 왜 문제인지:
- 수정 필요 여부:

5. 미검수 항목
확인하지 못한 항목이 있으면 반드시 적는다.
미검수 항목이 있으면 최종 PASS 금지.

6. 최종 수정 필요 목록
우선순위 순서로 작성한다.
```

---

## 6. 금지 문장

다음 문장은 금지한다.

- 전체적으로 깔끔합니다
- 문제 없어 보입니다
- PASS 가능해 보입니다
- 대체로 안전합니다
- 의도대로 보입니다
- 코드가 잘 정리되어 있습니다

위 표현을 쓰려면 실제 파일명, 함수명, selector명, 코드 구조 근거를 함께 써야 한다.

---

## 7. 화면 캡처 검수 원칙

UI는 코드 검수와 별도로 화면 검수가 필요하다.

코드상 PASS여도 화면에서 다음 문제가 있으면 UI PASS가 아니다.

- 카드가 사라져 텍스트만 떠 있음
- 카드 안 카드가 많음
- 색상 강조가 과함
- 글자 두께가 들쭉날쭉함
- 버튼이 화면 위계를 망침
- 상태값이 경고판처럼 보임
- line row가 clickable로 보이지 않음
- hover가 흔들림

브라우저 확인 전에는 UI 최종 PASS로 보지 않는다.

# Academy OS / APMS / EIE 공통 UI Interaction Guideline v1.1

> 적용 범위: Academy OS, APMS, EIE  
> 문서 상태: v1.1 기준 문서  
> 작성 기준일: 2026-05-29  
> 목적: 모달 중심 화면을 줄이고, 운영자·선생님·모바일 환경에 맞는 공통 UI 구조를 고정한다.

---

## 0. 배경

Academy OS / APMS / EIE는 기능이 커질수록 단순 팝업 중심 구조로는 유지보수가 어려워진다.

특히 다음 문제가 반복된다.

```text
- 모달 안의 모달
- 상세 보기용 팝업 남발
- 수정 중 화면 맥락 상실
- 목록과 상세의 관계 단절
- 모바일에서 작은 PC 화면을 억지로 축소
- 선생님 화면에서 현장 조작 속도 저하
```

따라서 앞으로 운영 UI는 다음 방향으로 통일한다.

```text
관리자 화면은 목록 + 인스펙터 패널.
선생님 화면은 카드 + 원터치 처리.
모바일은 바텀시트 또는 전용 화면.
모달은 최종 확인 전용.
```

---

## 1. 핵심 결론

기본 화면 흐름은 다음과 같다.

```text
목록 / 카드 / 표
→ 항목 선택
→ 우측 인스펙터 패널 또는 모바일 바텀시트
→ 상세 확인 / 수정 / 처리
→ 위험 작업만 최종 확인 모달
```

모달은 폐기하지 않는다.  
다만 역할을 **최종 확인 전용**으로 제한한다.

---

## 2. UI 선택 기준

## 2-1. 어떤 컴포넌트를 써야 하는가

| 상황 | 사용 컴포넌트 | 비고 |
|---|---|---|
| 단순 상세 확인 | InspectorPanel / BottomSheetPanel | 모달 금지 |
| 일반 수정 | InspectorPanel / BottomSheetPanel | 저장/취소는 패널 하단 |
| 메모 작성 | 패널 또는 인라인 | 가능하면 자동저장 |
| 상담 기록 확인 | 패널 탭 | 모달 금지 |
| 학생 정보 확인 | 패널 | 목록 맥락 유지 |
| 시간표 셀 확인 | 패널 | 배정 학생/수업 정보 표시 |
| 간단한 등록 | 패널 1페이지 폼 | 길어지면 전용 화면 |
| 긴 등록 | 전용 Wizard 화면 | 최종 확인만 모달 |
| 삭제 / 환불 / 권한 회수 | ConfirmModal | 위험도별 패턴 적용 |
| 대량 변경 | BulkActionBar + 미리보기 + ConfirmModal | 영향 범위 표시 |

---

## 2-2. 모달 사용 체크리스트

모달을 쓰기 전 반드시 다음 3가지를 확인한다.

```text
1. 데이터 유실 또는 손상의 위험이 있는가?
2. 사용자의 주의를 100% 집중시켜야 하는가?
3. 지금 작업 흐름을 끊어도 안전한가?
```

셋 중 하나라도 아니면 모달을 쓰지 않는다.  
대신 패널, 바텀시트, 인라인 경고, 토스트를 사용한다.

---

## 3. 화면 유형별 기본 구조

## 3-1. 관리자 / 원장 / 운영자 화면

관리자 화면은 정보량이 많고, 연속 확인·수정·처리가 중요하다.

기본 구조:

```text
왼쪽:
- 학생 목록
- 학원 목록
- 수업 목록
- 청구 목록
- 시간표 셀 목록
- 검색 / 필터 / 다중 선택

오른쪽:
- 선택한 대상의 상세 정보
- 수정 폼
- 처리 버튼
- 이력
- 메모
- 위험 작업 영역
```

관리자 화면에서는 페이지 이동이나 중앙 모달보다 **목록 + 우측 인스펙터 패널**을 우선한다.

---

## 3-2. 선생님 / 수업 운영 화면

선생님 화면은 현장 조작 속도가 가장 중요하다.

기본 구조:

```text
수업 카드
→ 학생 카드
→ 출석 / 결석 / 지각 / 숙제확인 / 교재진도
→ 1tap 처리
→ 즉시 반영 또는 빠른 저장
```

선생님 화면에서는 다음을 우선한다.

```text
- 큰 터치 영역
- 학생별 빠른 상태 변경
- 실시간 저장 느낌
- 실패 시 재시도 안내
- 불필요한 화면 이동 최소화
```

세부 기능은 카드 우측 상단 `⋮` 메뉴 또는 상세 패널로 숨긴다.  
자주 쓰는 액션 3~4개만 크게 노출한다.

---

## 3-3. 모바일 화면

모바일에서는 우측 패널을 그대로 쓰지 않는다.

| 화면 유형 | 데스크톱 | 모바일 |
|---|---|---|
| 관리자 상세 | 우측 인스펙터 패널 | 바텀시트 70~90% 또는 전체 화면 |
| 선생님 수업 | 카드형 화면 | 카드형 유지 + 하단 액션 |
| 긴 등록 | 전용 화면 | 전용 화면 |
| 위험 확인 | 확인 모달 | 확인 모달 |

## 5. InspectorPanel 원칙

## 5-1. 역할

InspectorPanel은 선택된 대상의 상세 정보와 처리 기능을 제공한다.

주요 용도:

```text
- 상세 정보 확인
- 정보 수정
- 이력 확인
- 메모 작성
- 상태 변경
- 수업 배정 확인
- 결제 / 청구 내역 확인
- 위험 작업 진입
```

---

## 5-2. 기본 구성

모든 InspectorPanel은 다음 구조를 따른다.

```text
상단 헤더
- 대상 이름
- 보조 정보
- 닫기 버튼
- Dirty State 배지

본문
- 기본 정보
- 상세 섹션
- 탭 또는 아코디언
- 이력 / 메모 / 관련 데이터

하단 액션 바
- 저장
- 취소
- 주요 처리 버튼
```

---

## 5-3. 너비와 레이아웃 기준

데스크톱 기본 레이아웃은 고정 비율보다 **목록 최소 폭 + 패널 권장 폭**을 우선한다.

```text
목록 최소 너비: 320px
목록 권장 너비: 320~420px
패널 최소 너비: 480px
패널 기본 너비: 520~640px
패널 최대 너비: 720px
```

권장 구현:

```css
grid-template-columns: minmax(320px, 420px) minmax(480px, 1fr);
```

정보량이 많은 화면은 50:50까지 허용한다.  
단, 패널이 화면 대부분을 차지해야 한다면 전용 화면으로 분리한다.

---

## 5-4. 브레이크포인트 기준

```text
>= 960px:
- 좌측 목록 + 우측 InspectorPanel 사용

< 960px:
- BottomSheetPanel 또는 전체 화면 패널 사용

< 768px:
- 기본은 BottomSheetPanel
- 긴 입력 플로우는 전용 화면
```

태블릿 경계에서는 목록이 320px 미만으로 줄어들면 즉시 바텀시트로 전환한다.

---

## 5-5. 패널 계층 제한

패널 안에서 또 패널을 여는 구조는 금지한다.

허용 구조:

```text
목록
→ 1단계 인스펙터 패널
→ 패널 내부 탭
```

금지 구조:

```text
목록
→ 패널
→ 패널
→ 패널
```

3단계 이상의 정보 구조가 필요하면 전용 화면으로 분리한다.

---

## 5-6. 패널 내부 탭

정보량이 많을 때는 패널 내부에 스티키 탭을 사용한다.

예시:

```text
학생 상세 패널
- 기본정보
- 수업
- 출결
- 숙제
- 상담
- 결제
- 이력
```

패널 내부 탭은 콘텐츠만 교체한다.  
패널 자체를 중첩해서 새로 열지 않는다.

---

## 6. 패널 열림 / 닫힘 규칙

## 6-1. 열림 기준

패널은 다음 동작에서 열린다.

```text
- 목록 행 클릭
- 카드 클릭
- 시간표 셀 클릭
- 검색 결과 클릭
```

선택된 항목이 바뀌면 기존 패널을 닫았다가 다시 여는 것이 아니라, 같은 패널 안에서 내용을 교체한다.

---

## 6-2. 닫힘 기준

패널은 다음 방식으로 닫을 수 있다.

```text
- X 버튼
- ESC 키
- 명시적 닫기 버튼
- 모바일 바텀시트 스와이프 다운
```

배경 클릭 닫기는 화면 성격에 따라 다르게 적용한다.

---

## 6-3. 배경 클릭 규칙

| 화면 성격 | 배경 클릭 |
|---|---|
| 목록을 보면서 계속 선택해야 하는 화면 | 허용 |
| 시간표 매칭 / 학생 배정 | 허용 |
| 민감 개인정보 수정 | 제한 |
| 환불 / 권한 / 삭제 | 차단 |
| 저장 전 변경사항 있음 | 확인 필요 |

EIE 시간표처럼 좌측 목록과 우측 패널을 함께 보며 작업해야 하는 화면은 배경을 흐리게 막지 않는다.  
반대로 위험 작업이나 민감정보 수정은 배경을 막는다.

---

## 7. Dirty State 처리 원칙

## 7-1. 변경 감지

패널 안에서 사용자가 값을 수정하면 Dirty State를 감지한다.

Dirty State가 있는 경우 패널 상단 또는 하단에 다음 상태를 표시한다.

```text
변경사항 있음
저장
취소
```

---

## 7-2. 수정 중 다른 항목 선택

수정 중 다른 목록 항목을 클릭하면 즉시 이동하지 않는다.

기본 처리 순서:

```text
1. 현재 패널에 변경사항 있음 표시
2. 저장 / 취소 / 계속 편집 선택 제공
3. 사용자가 선택한 뒤 다른 항목으로 이동
```

가능하면 인라인 배너로 처리한다.  
닫기, 메뉴 이동, 뒤로가기처럼 맥락이 크게 바뀌는 경우에만 ConfirmModal을 허용한다.

---

## 7-3. 패널 내부 탭 전환 시 Dirty State

탭 전환은 패널 닫힘으로 보지 않는다.

기본 원칙:

```text
- 같은 대상 안의 탭 전환은 허용
- 수정 중인 값은 PanelDraftStore에 유지
- DirtyStateBanner는 패널 전체 상단 또는 하단에 계속 표시
- 저장 / 취소는 패널 단위로 유지
```

단, 탭 전환 시 현재 탭 컴포넌트가 언마운트되면서 수정값이 사라지는 구조는 금지한다.  
구현상 보존이 불가능하면 탭 전환 전에 인라인 경고를 표시한다.

문구 기준:

```text
수정 중인 내용이 있습니다.
저장하거나 취소한 뒤 다른 탭으로 이동할 수 있어요.
```

---

## 7-4. 저장하지 않고 닫기

다음 상황에서는 최종 확인 모달을 허용한다.

```text
- X 버튼
- ESC 키
- 배경 클릭
- 뒤로가기
- 다른 메뉴 이동
```

문구 기준:

```text
수정 중인 내용이 있습니다.
저장하지 않고 닫을까요?
```

버튼 기준:

```text
계속 수정
저장하지 않고 닫기
```

---

## 8. 검색 / 필터 중 패널 동작

## 8-1. 필터 변경 후 선택 항목이 남아 있는 경우

현재 선택 항목이 필터 결과에 남아 있으면 패널을 유지한다.

```text
필터 변경
→ 목록 갱신
→ 선택 항목 유지
→ 패널 유지
```

---

## 8-2. 필터 변경 후 선택 항목이 사라진 경우

현재 선택 항목이 필터 결과에서 사라져도 즉시 패널을 닫지 않는다.

기본 처리:

```text
필터 변경
→ 목록에서 선택 항목 사라짐
→ 패널은 유지
→ 패널 상단에 “현재 필터 결과 밖 항목” 배지 표시
```

사용자가 다른 항목을 선택하거나 패널을 닫으면 자연스럽게 해제한다.

---

## 8-3. 항목이 삭제되었거나 권한이 사라진 경우

선택 항목을 더 이상 불러올 수 없으면 패널에 오류 상태를 표시한다.

문구 기준:

```text
이 항목을 불러올 수 없습니다.
삭제되었거나 접근 권한이 변경됐을 수 있어요.
```

---

## 9. 권한별 패널 접근 제어

## 9-1. 원칙

권한 조건은 화면 파일마다 흩어놓지 않는다.

다음 중 하나로 중앙화한다.

```text
PermissionGuard
uiPermissions
capability map
panel tab config
```

---

## 9-2. 탭 / 액션 노출 기준

패널 탭과 액션은 선언형 config로 관리한다.

예시:

```text
학생 상세 패널
- 기본정보: owner, manager, teacher
- 연락처: owner, manager
- 수업: owner, manager, teacher
- 결제: owner, manager
- 이력: owner, manager
- 삭제: owner only
```

프론트의 숨김 처리는 UX용이다.  
실제 보안은 서버 권한 검증으로 반드시 보장한다.

---

## 10. 저장 / 피드백 원칙

## 10-1. 저장 방식

| 작업 유형 | 저장 방식 |
|---|---|
| 메모 | 자동저장 |
| 체크박스 / 상태 토글 | 즉시 반영 |
| 학생 기본정보 수정 | 저장 버튼 |
| 청구 / 환불 / 권한 | 명시적 저장 + 확인 |
| 대량 변경 | 미리보기 + 확인 |

---

## 10-2. 상태 피드백 통일 문구

운영자·선생님 화면 모두 같은 짧은 메시지를 쓴다.

| 상태 | 문구 |
|---|---|
| 저장 중 | 저장 중 |
| 저장 완료 | 저장됨 |
| 실패 | 저장 실패 |
| 재시도 중 | 재시도 중 |
| 자동 재시도 예정 | 연결되면 다시 시도 |
| 변경 감지 | 변경사항 있음 |

불필요하게 긴 문구를 피한다.  
단, 실패 상태는 사용자가 다음 행동을 알 수 있게 한다.

---

## 10-3. 피드백 방식

모달 대신 가벼운 피드백을 우선한다.

```text
- 토스트
- 인라인 메시지
- 상태 배지
- 저장 중 표시
- 저장 완료 표시
- 실패 시 재시도 버튼
```

예시:

```text
저장됨
변경사항을 반영했어요
저장 실패 · 다시 시도
연결되면 다시 시도
```

---

## 11. 로딩 / 네트워크 상태 원칙

## 11-1. 로딩 표시

모든 비동기 작업에는 상태 표시가 있어야 한다.

```text
- 목록 로딩: Skeleton 우선
- 패널 상세 로딩: Skeleton 우선
- 짧은 액션 저장: 버튼 내부 저장 중
- 긴 처리: 인라인 진행 상태
```

기준:

```text
300ms 이상 걸릴 수 있는 데이터 로딩은 Skeleton 또는 상태 배지를 표시한다.
```

---

## 11-2. 선생님 화면 네트워크 원칙

선생님 화면은 교실 환경에서 네트워크가 불안정할 수 있다.

따라서 다음 원칙을 따른다.

```text
- 출결 / 숙제 / 교재 체크는 즉시 화면에 반영
- 서버 저장은 뒤에서 처리
- 실패 시 상태를 되돌리지 않고 실패 배지 표시
- 재시도 큐에 넣기
- 사용자가 같은 버튼을 여러 번 누르지 않게 처리
```

상태 예시:

```text
저장 중
저장됨
재시도 중
저장 실패
```

---

## 12. 다중 선택 / Bulk Action 원칙

## 12-1. 진입 방식

관리자 화면에서는 다중 선택을 지원한다.

기본 구조:

```text
좌측 목록 체크박스 선택
→ Bulk Action Bar 표시
→ 선택된 항목 수 표시
→ 일괄 처리
```

---

## 12-2. 다중 선택 중 단일 항목 클릭

다중 선택 상태에서는 row 클릭 동작을 바꾼다.

```text
선택 항목 0개:
- row 클릭 = 상세 패널 열기

선택 항목 1개 이상:
- row 클릭 = 선택 토글
- 상세 열기는 별도 “상세” 버튼 또는 이름 클릭으로 제한
```

이 기준을 쓰면 대량 작업 중 실수로 패널이 열리는 문제를 줄일 수 있다.

---

## 12-3. 영향 범위 표시

대량 처리 전 반드시 영향 범위를 표시한다.

예시:

```text
12명 선택됨
- 처리 대상: 9명
- 이미 비활성화: 2명
- 권한 부족: 1명
```

위험한 대량 변경은 ConfirmModal 전에 미리보기 화면 또는 인라인 요약을 거친다.

---

## 13. Empty State 원칙

아무 항목도 선택하지 않은 경우 우측 패널 영역을 비워두지 않는다.  
반드시 의미 있는 Empty State를 노출한다.

톤은 짧고 친절하게 유지한다.

예시:

```text
학생을 선택하면 상세 정보가 여기에 표시됩니다.
수업을 선택하면 배정 학생과 수업 정보를 볼 수 있습니다.
청구 항목을 선택하면 수납 상태와 처리 내역을 볼 수 있습니다.
```

너무 긴 설명은 피한다.

---

## 14. DangerZone 원칙

## 14-1. 위치

DangerZone은 패널 하단에 배치한다.  
일반 액션과 섞지 않는다.

---

## 14-2. 시각 기준

```text
- 상단 구분선
- 빨간 계열 경고 색상
- 경고 아이콘 또는 위험 배지
- 짧은 설명
- 보조 버튼 형태의 위험 액션
```

삭제, 환불, 권한 회수, 초기화는 반드시 DangerZone 또는 전용 위험 플로우에서만 진입한다.

---

## 15. ConfirmModal 위험도 패턴

| 위험도 | 예시 | 확인 방식 |
|---|---|---|
| 낮음 | 단순 상태 변경 취소 | 확인 / 취소 |
| 중간 | 학생 비활성화, 수업 숨김 | 사유 선택 |
| 높음 | 환불, 권한 회수, 대량 변경 | 사유 선택 + 영향 범위 표시 |
| 매우 높음 | 데이터 삭제, 초기화 | 대상명 직접 입력 |

ConfirmModal은 일반 상세 보기나 수정에 사용하지 않는다.

---

## 16. 키보드 / 접근성 원칙

## 16-1. 기본 접근성

공통 패널은 다음 접근성 기준을 따른다.

```text
- ESC 키로 닫기
- 닫기 버튼 제공
- 키보드 포커스 순서 유지
- 저장 버튼 접근 가능
- 위험 버튼은 시각적으로 분리
- 스크린리더용 제목 구조 유지
```

---

## 16-2. 초기 필수 단축키

관리자 화면에서는 다음 3개를 초기부터 우선 지원한다.

| 단축키 | 동작 |
|---|---|
| ESC | 패널 닫기 또는 현재 동작 취소 |
| E | 선택 항목 편집 진입 |
| Ctrl + S | 저장 |

추가 도입 후보:

| 단축키 | 동작 |
|---|---|
| / | 검색 포커스 |
| Del | 삭제 확인 열기 |
| ↑ / ↓ | 목록 이동 |

단축키는 input, textarea, contenteditable 입력 중에는 방해하지 않는다.

---

## 17. 시간표 Drag & Drop 중 패널 동작

시간표 화면은 드래그와 패널 선택이 충돌하기 쉽다.

기본 원칙:

```text
드래그 시작:
- 기존 패널은 유지
- 패널 내용 자동 전환 금지

드래그 중:
- 선택 상태 변경 금지
- 충돌 가능성은 인라인으로 표시

드롭 성공:
- 관련 셀 데이터 갱신
- 열려 있는 패널 대상이 영향을 받으면 패널 내용도 갱신

드롭 실패:
- 원래 위치로 되돌림
- 실패 사유를 인라인 또는 토스트로 표시
```

대량 이동, 기존 배정 제거, 충돌 무시 같은 위험 작업은 ConfirmModal을 거친다.

---

## 18. 디자인 토큰 / 시각 원칙

## 18-1. 액션 색상

```text
Primary Action:
- 저장, 적용, 확인

Secondary Action:
- 취소, 닫기, 뒤로

Danger Action:
- 삭제, 환불, 권한 회수, 초기화
```

Danger Action은 Primary Action과 같은 위치·색상·강도로 배치하지 않는다.

---

## 18-2. 저장 버튼 위치

저장 버튼은 가능한 한 패널 하단 우측에 고정한다.  
변경사항이 있는 상태에서는 저장 버튼 또는 DirtyStateBanner를 명확히 강조한다.

---

## 19. 레거시 모달 마이그레이션 기준

기존 모달은 다음 기준으로 분류한다.

| 기존 모달 용도 | 처리 |
|---|---|
| 단순 상세 보기 | 패널로 전환 |
| 일반 수정 | 패널로 전환 |
| 학생 정보 보기 | 패널로 전환 |
| 수업 정보 보기 | 패널로 전환 |
| 메모 작성 | 패널 또는 인라인 |
| 삭제 확인 | ConfirmModal 유지 |
| 환불 확인 | ConfirmModal 유지 |
| 권한 회수 | ConfirmModal 유지 |
| 대량 처리 | 전용 플로우 + ConfirmModal |

## 20. 화면별 적용 우선순위

## 20-1. 1순위: EIE 학생관리

가장 먼저 적용한다.

이유:

```text
- 학생 목록 + 상세 구조가 명확함
- 연락처 / 수업 배정 / 상태 관리가 늘어날 예정
- 모달로 가면 빠르게 복잡해짐
```

구조:

```text
왼쪽:
학생 목록 / 검색 / 상태 필터

오른쪽:
학생 상세 패널
- 기본정보
- 연락처
- 수업 배정
- 상담 / 메모
- 이력
```

---

## 20-2. 2순위: EIE 시간표 / 클래스룸

EIE 시간표는 수업 셀 중심 구조다.

구조:

```text
시간표 셀 클릭
→ 우측 패널
→ 수업 정보
→ 배정 학생
→ 학생 추가 / 제외 / 이동
```

클래스룸은 선생님 화면에 가까우므로 카드형을 유지한다.

---

## 20-3. 3순위: 청구 / 결제 관리

청구와 결제는 위험 작업이 많다.

구조:

```text
청구 목록
→ 청구 상세 패널
→ 수납 처리
→ 환불 / 취소 / 정정은 ConfirmModal
```

---

## 20-4. 4순위: 선생님 화면 리뉴얼

선생님 화면은 패널보다 카드형 최적화가 우선이다.

구조:

```text
오늘 수업
→ 수업 카드
→ 학생 카드
→ 출결 / 숙제 / 교재 / 메모
```

## 23. QA 검증 시나리오

## 23-1. 학생관리 기본

```text
1. 학생 목록에서 학생 클릭
2. 우측 패널이 열린다
3. 중앙 모달은 뜨지 않는다
4. 다른 학생 클릭 시 같은 패널 안에서 내용이 바뀐다
```

---

## 23-2. Dirty State

```text
1. 학생 상세 패널에서 이름 수정
2. 저장하지 않고 다른 학생 클릭
3. DirtyStateBanner가 보인다
4. 저장 / 취소 / 계속 편집 선택이 가능하다
```

---

## 23-3. 패널 내부 탭

```text
1. 기본정보 탭에서 수정
2. 수업 탭 클릭
3. 수정값이 사라지지 않는다
4. 변경사항 있음 상태가 유지된다
```

---

## 23-4. 검색 / 필터

```text
1. 학생 상세 패널을 연다
2. 검색어 또는 필터를 바꾼다
3. 선택 항목이 결과에서 사라져도 패널이 즉시 닫히지 않는다
4. 현재 필터 결과 밖 항목 상태가 표시된다
```

---

## 23-5. 위험 작업

```text
1. 학생 삭제 버튼 클릭
2. ConfirmModal이 열린다
3. 위험 문구와 영향 범위가 표시된다
4. 필요한 경우 대상명 직접 입력 후 확인 가능하다
```

---

## 23-6. 모바일

```text
1. 960px 이하 화면에서 학생 클릭
2. 우측 패널이 아니라 BottomSheetPanel로 열린다
3. 768px 이하에서는 하단 카드 또는 전체 화면으로 자연스럽게 전환된다
```

## 24. 금지 패턴

앞으로 다음 패턴은 지양한다.

```text
- 모달 안의 모달
- 패널 안의 패널
- 상세 보기용 중앙 팝업
- 수정용 대형 모달
- 닫기 기준이 없는 패널
- 저장 실패 표시 없는 원터치 처리
- 선택 항목 없는 빈 우측 영역
- 위험 작업을 일반 버튼처럼 배치
- 모바일에서 작은 PC용 표를 그대로 축소
- 권한 조건을 화면 파일마다 흩어놓기
- Dirty State가 있는 탭을 언마운트해서 수정값 날리기
```

## 25. 최종 기준 문장

Academy OS, APMS, EIE의 운영 UI는 앞으로 다음 기준을 따른다.

```text
관리자 화면은 목록 + 인스펙터 패널.
선생님 화면은 카드 + 원터치 처리.
모바일은 바텀시트 또는 전용 화면.
모달은 최종 확인 전용.
패널은 1단계까지만.
권한은 중앙 config로 제어.
위험 작업은 명확하게 분리.
저장 상태와 실패 상태는 반드시 표시.
```

이 기준을 공통 디자인 시스템과 개발 가이드라인에 고정한다.

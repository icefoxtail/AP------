기출 데이터 무결성 성역화 프로토콜 V10.5 [JS아카이브 정합화판 / 제한적 임의보정 허용]

본 프로토콜은 시험지 이미지 또는 PDF 원본으로부터 문항을 추출할 때,
AI의 자의적 판단, 의역, 요약, 문장 다듬기, 구조 변경을 기본적으로 금지하되,
실무상 불가피한 판독 한계가 있는 경우에는 제한적 임의보정을 허용하는 버전이다.

본 프로토콜은 JS아카이브 프로젝트의 실제 운영 기준에 맞춘 추출용 프로토콜이다.
검수 프로토콜이 아니며, 수정 실행 프로토콜도 아니다.
목적은 원본 문항을 엔진 호환 구조로 최대한 안전하게 복원하는 것이다.

--------------------------------------------------
0. 프로토콜 성격
--------------------------------------------------

- 이 문서는 원본 시험지 추출용이다.
- 검수 프로토콜이 아니다.
- 수정 실행 프로토콜도 아니다.
- 목적은 원본 문항을 엔진 호환 구조로 안전하게 복원하는 것이다.
- 추출 단계에서는 원문 우선 원칙을 유지한다.
- 다만 지정된 예외 상황에서는 제한적 임의보정을 허용한다.
- 검수와 수정은 추출 이후 별도 단계에서 수행한다.

--------------------------------------------------
1. 최상위 원칙
--------------------------------------------------

[1-1. 원문 우선]
- 발문, 보기, 정답, 표, 도형, 그래프, 조건문은 원본 우선으로 추출한다.
- 원문을 더 자연스럽게 다듬거나 문장을 재구성하지 않는다.
- 문장 순서, 조건 순서, 보기 순서를 임의 변경하지 않는다.

[1-2. 학생 필기 완전 배제]
- 학생이 쓴 손글씨, 체크, 동그라미, 계산 흔적, 메모, 낙서는 원본 문제가 아니다.
- 학생 필기를 단 한 글자도 추출에 포함하지 않는다.
- 학생 필기와 인쇄 원본이 겹쳐도, 가능한 한 인쇄 원본만 복원 대상으로 본다.

[1-3. 제한적 임의보정 허용]
- 아래 예외 상황에서는 작업 중단 대신 제한적 임의보정을 허용한다.
- 단, 임의보정은 최소 범위에서만 수행한다.
- 임의보정한 항목은 최종 출력 뒤에 반드시 별도 보고한다.
- 임의보정은 “가능성이 높은 최소 보정”만 허용되며, 문제 전체를 재창작하는 수준으로 확장하면 안 된다.

[1-4. 검수·수정 분리]
- 추출 단계에서 검수 결과를 덮어쓰지 않는다.
- 추출 단계에서 해설을 억지로 맞추지 않는다.
- 수리 검토와 수정은 별도 검수/수정 프로토콜에서 수행한다.

[1-5. 엔진 호환 우선]
- 최종 산출물은 JS아카이브 엔진이 읽을 수 있는 구조여야 한다.
- 최종 목표는 window.questionBank 호환 데이터다.
- 추출 단계에서 구조를 깨뜨리면 실패다.

--------------------------------------------------
2. 작업 순서 [EXECUTION ORDER]
--------------------------------------------------

문항마다 반드시 아래 순서를 따른다.

STEP 1. 이미지 / PDF 정밀 판독
STEP 2. 학생 필기 제거 및 레이아웃 판단
STEP 3. 발문(content) 추출
STEP 4. 보기(choices) 추출
STEP 5. 정답(answer) 확정 또는 정답불가 표기
STEP 6. 표/도형/그래프/좌표평면 재현 필요 여부 판정
STEP 7. FINAL RENDER 형태로 우선 출력
STEP 8. 사용자 검수 완료 후에만 MACHINE OUTPUT(JS/JSON) 생성
STEP 9. 제한적 임의보정 내역 최종 보고

순서를 바꾸거나 일부 단계를 건너뛰면 프로토콜 위반이다.

--------------------------------------------------
3. 학생 필기 완전 배제 [CONTAMINATION FILTER]
--------------------------------------------------

배제 대상:
- 손글씨 정답 표시
- 동그라미, 체크표시, 별표, X표
- 여백 계산식, 풀이 메모, 숫자 적기
- 형광펜, 수정액, 지우개 흔적
- 문제 아래/옆의 임의 낙서

판단 기준:
- 인쇄된 폰트, 선, 표, 도형만 원본으로 인정한다.
- 손글씨는 무조건 배제한다.
- 손글씨와 인쇄가 섞인 경우에도 가능한 범위에서 인쇄 원본만 복원한다.
- 복원이 애매할 경우에도 작업 중단하지 않고, 제한적 임의보정 규칙으로 넘어간다.

--------------------------------------------------
4. 다단 레이아웃 처리 [MULTI-COLUMN PROTOCOL]
--------------------------------------------------

- 먼저 전체 페이지가 1단인지 2단인지 판단한다.
- 2단이면 반드시 좌측 단을 먼저 끝낸 후 우측 단으로 이동한다.
- 좌측 발문에 우측 수식이 섞이거나, 우측 보기 일부가 좌측 문제에 붙는 식의 추출은 금지한다.
- 컬럼 경계가 불명확한 경우에도 가능한 범위에서 문항 단위로 분리하여 처리하되, 필요한 경우 제한적 임의보정 보고에 포함한다.

--------------------------------------------------
5. 제한적 임의보정 허용 규칙
--------------------------------------------------

아래 경우에는 작업 중단 대신 임의보정을 허용한다.

허용 상황:
- 해상도가 낮아 글자 형태를 정확히 알기 어려운 경우
- i 와 1, α 와 a, β 와 B, θ 와 0 등 혼동 가능 기호가 있는 경우
- 지수 ², ³, 첨자, 분수선, 루트 기호가 불명확한 경우
- 보기 일부가 잘렸거나 번진 경우
- 표의 숫자, 도형의 점, 각도, 길이 표기가 불명확한 경우
- 원문 조건이 모호하여 직접 풀이에도 최소 가정이 필요한 경우

임의보정 원칙:
- 가장 작은 범위만 보정한다.
- 문맥, 수식 구조, 보기 구조, 직접 풀이 결과를 함께 참고한다.
- 정답이 맞아야 하므로 판독 결과를 억지로 끼워 맞추는 것은 금지하지만,
  문항 전체 구조상 불가피한 최소 보정은 허용한다.
- 보정 후에도 발문 구조, 보기 구조, 문장 골격은 가능한 유지한다.
- 원문 전체를 다시 쓰거나 창작하는 수준의 변경은 금지한다.

임의보정 예시:
- 보기 ⑤의 `$` 시작/끝 누락 보정
- `i`와 `1` 중 하나가 명백히 식 구조상 맞는 경우 선택 보정
- 지수 ²가 번졌지만 계산 구조상 제곱으로만 성립하는 경우 보정
- 도형의 점 이름이 흐리지만 동일 문항 내 다른 표기와 일치하는 쪽으로 보정
- 표 셀 숫자가 일부 흐리지만 행/열 규칙상 유일하게 복원 가능한 경우 보정

금지되는 임의보정:
- 발문 뜻 자체를 바꾸는 보정
- 보기 5개를 새로 창작하는 보정
- 정답에 맞추기 위해 조건을 뜯어고치는 보정
- 원문보다 더 좋은 문항이 되도록 다듬는 보정
- 교육과정 메타를 임의 재분류하는 보정

--------------------------------------------------
6. CHECKPOINT
--------------------------------------------------

━━ CHECKPOINT 1: 발문(content) ━━
- 발문 전체를 추출한 직후 원본과 최소 2회 대조한다.
- 수식, 조건문, 단위, 괄호, 줄바꿈, 표 앞뒤 문장을 다시 본다.
- 표나 도형이 있으면 발문 바로 아래에 올 요소가 누락되지 않았는지 본다.
- 임의보정이 들어간 부분이 있으면 별도 메모한다.

━━ CHECKPOINT 2: 보기(choices) ━━
- 객관식이면 ①~⑤를 각각 1개씩 대조한다.
- 보기 5개가 정확히 있는지 본다.
- 보기 내부 ㄱ, ㄴ, ㄷ 조건이 누락되지 않았는지 본다.
- i, α, β, 지수, 부등호가 틀리지 않았는지 본다.
- 필요한 경우 최소 범위 임의보정을 허용하되 기록한다.

━━ CHECKPOINT 3: 정답(answer) ━━
- 직접 풀어서 정답을 확인한다.
- 풀이 결과가 보기 중 하나와 정확히 일치하는지 확인한다.
- 정답이 명확하지 않으면 정답불가로 처리할 수 있다.
- 임의보정이 들어간 경우, 보정 전후 판단 근거를 메모한다.

CHECKPOINT 3개를 모두 통과한 문항만 FINAL RENDER에 포함한다.

--------------------------------------------------
7. FINAL RENDER / MACHINE OUTPUT 분리
--------------------------------------------------

[7-1. FINAL RENDER]
- 사용자가 원문 누락 여부를 확인할 수 있도록 시험지 형태로 우선 출력한다.
- 이 단계에서는 solution을 붙이지 않는다.
- 발문, 보기, 표, 도형, 그래프, 조건만 원문 기준으로 보여준다.
- 단, 제한적 임의보정이 들어간 경우에는 본문은 복원본 그대로 출력하고,
  보정 내역은 본문 바깥 최종 보고에서만 알린다.

[7-2. MACHINE OUTPUT]
- 사용자의 검수가 끝난 뒤에만 생성한다.
- JS아카이브 엔진 호환 구조로 출력한다.
- 기본 목표는 아래 구조다.

window.examTitle = "파일명형식";
window.questionBank = [
  {
    id: 1,
    level: "중",
    category: "...",
    originalCategory: "...",
    standardCourse: "...",
    standardUnitKey: "...",
    standardUnit: "...",
    standardUnitOrder: 1,
    questionType: "객관식",
    layoutTag: "grid",
    tags: [],
    wide: false,
    content: "...",
    choices: ["① ...", "② ...", "③ ...", "④ ...", "⑤ ..."],
    answer: "③",
    solution: "..."
  }
];

--------------------------------------------------
8. 텍스트 복제 원칙 [ZERO ALTERATION]
--------------------------------------------------

- 원문 텍스트는 한 글자 단위로 최대한 동일하게 유지한다.
- 띄어쓰기, 줄바꿈, 괄호, 조건 순서, 보기 순서 임의 변경 금지
- 오타가 있어도 본문은 함부로 고치지 않는다.
- 오타 여부는 검수 단계에서 별도로 판정한다.
- 추출 단계에서는 먼저 원문 복원이 우선이다.
- 단, 5장에서 허용한 제한적 임의보정 상황은 예외로 한다.

--------------------------------------------------
9. 수식 표기 원칙 [JS아카이브 엔진 기준]
--------------------------------------------------

- MACHINE OUTPUT 단계에서는 룰북 수식 규칙을 따른다.
- 수식은 필요한 부분만 $...$ 사용
- JS 문자열 내부 LaTeX 백슬래시는 반드시 이중 처리
- 줄바꿈은 \\n 사용
- $...$는 \\n 기준 각 줄마다 닫아야 한다
- choices 안의 부등호는 \\lt, \\gt 사용
- 한국어를 \\text{} 안에 넣지 않는다
- solution 안 영어 therefore, hence, implies 사용 금지
- 대신 $\\therefore$, $\\implies$, $\\because$ 사용
- 제한적 임의보정으로 수식 기호를 보정한 경우 최종 보고에 남긴다

--------------------------------------------------
10. 표 / 도형 / 그래프 / SVG 처리
--------------------------------------------------

[10-1. 공통 원칙]
- 발문에서 표, 그림, 도형, 그래프가 언급되면 추출 대상에 반드시 포함한다.
- 발문 바로 아래 배치 원칙을 지킨다.
- 표와 도형은 누락 없이 함께 복원해야 한다.

[10-2. 표]
- 칸 구조, 병합, 숫자, 정렬, 테두리를 최대한 원본대로 복원한다.
- MACHINE OUTPUT 단계에서는 HTML table로 출력한다.
- 발문 표는 `.question-table-wrap` 또는 `.question-table` 전략을 따르는 구조가 바람직하다.
- `<table>`, `<tr>`, `<td>`, `<th>` 직전·직후 `<br>` 금지
- 표 숫자가 불명확할 경우 최소 범위 임의보정을 허용하되 최종 보고에 남긴다.

[10-3. 도형 / 그래프]
- 정확한 재현이 필요한 경우 Python 기반 좌표 계산 후 SVG 또는 이미지로 만든다.
- 원본이 명확하면 선 길이, 각도, 점 위치, 눈금, 비율을 맞춘다.
- 일부 표기가 흐린 경우 최소 범위 임의보정을 허용한다.
- 도형 전체를 새로 창작하는 것은 금지한다.

[10-4. SVG 규칙]
- SVG 내부 `<br>` 금지
- SVG 내부 LaTeX($...$, \\frac, \\sqrt 등) 금지
- `<text>` font-size는 원칙적으로 8px / 9px / 10px 기준 사용
- 다중 줄 텍스트는 `<tspan>` 사용
- 실제 도형이 삽입되면 기존 `[도형필요]`, `[해설도형필요]` 마커는 삭제해야 한다

--------------------------------------------------
11. questionType / layoutTag 규칙
--------------------------------------------------

- questionType은 문항 성격 분류용이다.
  - "객관식" | "주관식" | "서술형"
- layoutTag는 렌더링 배치용이다.
  - "grid" | "subjective-2up" | "subjective-4up" | "fullwidth"

강제 원칙:
- questionType과 layoutTag는 분리한다.
- choices.length === 0만으로 특수배치를 추정하지 않는다.
- 주관식이라고 자동으로 B2/B4/fullwidth를 부여하지 않는다.
- 특수배치는 layoutTag 또는 tags로만 판정한다.
- wide:true는 사용자 수동 지정 전용이다.

--------------------------------------------------
12. 표준단원 메타데이터 원칙
--------------------------------------------------

- standardCourse, standardUnitKey, standardUnit, standardUnitOrder는 마스터 테이블 기준으로 채운다.
- 중학교는 M1 / M2 / M3 계열 사용
- 고등학교는 원본 기준에 따라 H15 또는 H22 사용 가능
- 2015 문항을 승인 없이 H22로 바꾸지 않는다
- 2022 문항을 승인 없이 H15로 바꾸지 않는다
- 대응 가능한 단원이 있는데 RAW를 쓰면 안 된다
- 메타데이터는 원문 판독과 별개로 프로젝트 기준에 맞게 채우되, 임의 추정 매핑은 금지한다

--------------------------------------------------
13. 출력 언어 및 사용자 노출 규칙
--------------------------------------------------

- 사용자에게는 자연어와 최종 출력만 보여준다.
- 내부 처리용 STEP, CHECKPOINT, JSON 설명을 장황하게 노출하지 않는다.
- 작업 중 검수 단계에서는 FINAL RENDER만 보여준다.
- 전체 검수 후에만 MACHINE OUTPUT을 생성한다.

--------------------------------------------------
14. 임의보정 보고 규칙 [V10.5 핵심]
--------------------------------------------------

- 제한적 임의보정이 1건이라도 들어갔으면 최종 출력 뒤에 반드시 보고한다.
- 본문 내부에 주석을 끼워 넣지 않는다.
- 별도 보고 섹션으로만 정리한다.

17. 최종 행동 명령
--------------------------------------------------

원본 이미지를 먼저 판독하라.

원문 복제와 엔진 호환을 동시에 만족시키는 것을 최종 목표로 삼아라.

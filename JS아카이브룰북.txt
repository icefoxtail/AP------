## 2. engine.html 아키텍처 (v1.6.1 기준)

### 2-1. 최상위 원칙
- 시험지 렌더링은 Single Sequence 방식으로 처리한다.
- 문항은 데이터 순서를 절대 유지한다.
- wide / subj / normal을 순차 판정하고, 뒤에서 재조합하지 않는다.
- 페이지 분리는 Safe Page-Break 규칙으로 처리한다.
- renderExam은 비성역이지만, 현재 v1.6.1 구조를 기준선으로 삼는다.

### 2-2. 성역 / 준성역 구분
성역:
- wrapLatex(text)
- autoCompress(container)
- fitQuestionBox(box)
- renderSol(area, data)
- renderAns(area, data)
- makePage(area, type, pNum)
- MathJax 호출 순서
- staging 구조

준성역:
- renderExam(area, data)
  단, 현행 v1.6.1 구조를 기준으로 최소 수정만 허용한다.

### 2-3. 번호 정책
- 시험지 번호는 엔진 q-num이 출력한다.
- renderExam의 문항 box 생성 시 아래 구조를 기본으로 한다.
  `<div class="q-num">${i + 1}.</div><div>${wrapLatex(q.content)}</div>${choicesHTML}`
- content 맨 앞에 문항번호(예: `1.`, `2.`)를 넣지 않는다.
- 엔진 번호와 content 번호가 동시에 존재하면 중복 오류로 간주한다.
- 번호 없는 데이터는 content 오류가 아니라 엔진 출력으로 보완된다.

### 2-4. 문항 유형 판정
- `wide === true` 또는 `layoutTag === "fullwidth"` → `wide`
- `layoutTag === "subjective-2up"` 또는 `tags.includes("서술형2")` → `subj2`
- `layoutTag === "subjective-4up"` 또는 `tags.includes("서술형4")` → `subj4`
- 나머지 → `normal`

주의:
- `choices.length === 0`만으로 특수배치 판정 금지
- `wide` 자동 판정 금지
- fullwidth는 사용자 수동 지정만 허용

### 2-5. 렌더링 구조
- renderExam은 staging에서 프록시 높이를 측정한 뒤 block을 순차 생성한다.
- normal 블록은 qpp 기준으로 chunk를 만든다.
- subj2는 2문항 1블록
- subj4는 4문항 1블록
- wide는 1문항 단독 블록
- block들은 pages 배열에 순차적으로 쌓고, 마지막에 pages를 렌더한다.

### 2-6. Safe Page-Break 규칙
- flushBlock 내부에서 강제 page push를 하지 않는다.
- pages.push(currentPage)는 반드시 `currentPage.blocks.length > 0`일 때만 수행한다.
- 특수문항(wide/subj) 전후 흐름 차단은 items 순회 중에만 결정한다.
- 빈 페이지 생성 금지
- 첫 페이지는 지연 생성한다.

### 2-7. 페이지별 배치 규칙
- normal 블록:
  - 2열 grid
  - `bGrid.style.flex = '1 1 0'`
  - 균등배분 대상
- subj2 / subj4:
  - 2열 grid
  - `bGrid.style.flex = 'none'`
  - 균등배분 대상 아님
- wide:
  - 단독 full-width
  - page.body에 wideCol 단독 삽입

### 2-8. 현재 미해결 이슈
- 객관식은 1페이지에서는 4분할이 되지만, 2페이지부터 동일한 4분할 균등배분이 깨지는 현상이 있다.
- 현재 엔진은 normal 블록에만 `flex: 1 1 0`을 주고 있으나, 페이지 전환 이후 block 누적/분리 조건 때문에 2페이지 이후 균등배분이 완전히 유지되지 않을 수 있다.
- 이 항목은 v1.6.1의 known issue로 기록한다.
- 룰북상 현재 상태는:
  - wide / subj 분리 유지
  - 번호 정책 복구 유지
  - 2페이지 이후 normal 4분할 균등배분은 후속 수정 대상

### 2-9. 금지 사항
- Group A/B/C 후행 렌더 구조로 회귀 금지
- fullwidth를 뒤로 자동 이동 금지
- 문항 순서 재정렬 금지
- 번호 정책을 다시 content 기준으로 되돌리기 금지
- q-num 제거 금지
- renderSol / renderAns 수정 금지

### 2-10. 운영 메모
- 현재 엔진 기준선은 Stage E v1.6.1이다.
- 다음 우선 수정은 “객관식 2페이지 이후 4분할 붕괴” 해결이다.
- 그 전까지는 wide/subj/번호정책은 고정하고, normal block의 page 2 이후 flex/height 처리만 집도한다.

## 3. JS 문항뱅크 데이터 규칙

### 3-1. 파일 헤더

```js
window.examTitle = "YY_학교_N학기_시험종류_학년_과목";
// 예: "24_효천고_1학기_중간_고1_수학(상)"
```

### 3-2. 문항 객체 구조

```js
{
  id: 1,                          // 1부터 순차 정수
  level: "중",                    // "하" | "중" | "상"
  category: "단원명",             // 출제 교재 기준 단원
  originalCategory: "원본단원명", // 원본 시험지 표기 그대로
  standardCourse: "수학(상)",     // 표준 과목명
  standardUnitKey: "H15-SA-01",   // 표준단원키
  standardUnit: "다항식",         // 표준단원명
  standardUnitOrder: 1,           // 표준단원 순서

  questionType: "객관식",         // "객관식" | "주관식" | "서술형" (선택 필드)
  layoutTag: "grid",              // "grid" | "subjective-2up" | "subjective-4up" | "fullwidth" (선택 필드)
  tags: [],                       // 예: ["서술형2"] (선택 필드)
  wide: false,                    // 사용자 수동 지정 예외 배치 플래그 (선택 필드)

  content: "발문 텍스트",         // 수식은 $...$로 감싸기
  choices: ["①", "②", "③", "④", "⑤"],  // 주관식은 []
  answer: "③",                   // 주관식 수식은 $...$ 필수
  solution: "[키포인트] ...\n풀이..."
}
```

### 3-3. 수식 작성 규칙 (Gemini 기준)

| 규칙                         | 올바른 예              | 틀린 예         |
| -------------------------- | ------------------ | ------------ |
| 수식은 `$...$`로 감싸기           | `$x^2 + 1$`        | `x^2 + 1`    |
| 줄바꿈은 `\\n` (JS 문자열 내)      | `"1단계\\n2단계"`      | `"1단계\n2단계"` |
| `$...$`는 `\\n` 기준 각 줄마다 닫기 | `"$a$\\n$b$"`      | `"$a\\nb$"`  |
| `\\n`으로 시작하는 명령어           | `$\\notin A$`      | `\notin A`   |
| 집합 표기                      | `$\\{x \| x>0\\}$` | `{x\|x>0}`   |
| choices 부등호                | `\\lt`, `\\gt`     | `<`, `>`     |
| `\text{}` 한국어 금지           | 평문으로 적기            | `\text{이상}`  |
| 조합 기호                      | `${}_{n}C_{r}$`    | `{}_nC_r`    |
| 주관식 answer 수식              | `"$\\frac{1}{2}$"` | `"1/2"`      |
| solution 시작                | `[키포인트] ...`       | 바로 풀이 시작     |

### 3-4. choices 형식

* **객관식**: `["①", "②", "③", "④", "⑤"]`
* **주관식**: `[]`
* 값에 수식 포함 시: `["① $x=1$", "② $x=2$", ...]`

### 3-5. solution 형식

```text
[키포인트] 이 문제의 핵심 개념 한 줄
1단계: ...
2단계: ...
∴ 정답: ③
```

## 5. 표준 교육과정 및 표준단원키 운영 규칙

### 5-1. 기준 원본
- 표준 교육과정, standardCourse, standardUnitKey, standardUnit, standardUnitOrder의 최종 기준 원본은
  `# JS아카이브 표준단원키 마스터 테이블.md` 로 본다.
- 룰북은 운영 원칙과 사용 규칙을 설명하는 문서이고,
  실제 단원명/키/순서의 기준 데이터는 반드시 마스터 테이블을 따른다.

### 5-2. 중학교 교육과정 기준
중학교는 2022 개정 기준으로 다음 체계를 사용한다.

- 중1: M1-01 ~ M1-08
- 중2: M2-01 ~ M2-08
- 중3: M3-01 ~ M3-07

중학교 standardCourse 표기 원칙:
- `중1 수학`
- `중2 수학`
- `중3 수학`

중학교 standardUnitKey / standardUnit / standardUnitOrder는
반드시 마스터 테이블의 값을 그대로 사용한다.

### 5-3. 고등학교 교육과정 기준
고등학교는 자료의 성격에 따라 2015 개정과 2022 개정 체계를 모두 보유할 수 있다.

#### 2015 개정
- H15-SA : 수학(상)
- H15-SB : 수학(하)
- H15-M1 : 수학 I
- H15-M2 : 수학 II
- H15-PS : 확률과 통계
- H15-GV : 기하와 벡터

#### 2022 개정
- H22-C : 공통수학 1
- H22-C2 : 공통수학 2
- H22-A : 대수
- H22-M : 미적분
- H22-PS : 확률과 통계
- H22-GE : 기하

### 5-4. standardCourse 표기 원칙
- 2015 개정 문항은 해당 과목명을 그대로 사용한다.
  - 예: `수학(상)`, `수학 I`, `수학 II`, `확률과 통계`
- 2022 개정 문항은 해당 과목명을 그대로 사용한다.
  - 예: `공통수학 1`, `공통수학 2`, `대수`, `미적분`, `확률과 통계`, `기하`

### 5-5. 매핑 원칙
- 문항의 실제 교육과정 기준에 맞는 standardCourse와 standardUnitKey를 사용한다.
- 2015 개정 문항을 임의로 H22 계열로 바꾸지 않는다.
- 2022 개정 문항을 임의로 H15 계열로 바꾸지 않는다.
- 과목명만 보고 자동 치환하지 않고,
  문항 출처와 실제 교육과정 맥락을 함께 확인한다.

### 5-6. RAW 사용 원칙
- 마스터 테이블에 즉시 대응 단원이 없을 때만 `RAW-단원명` 형식을 임시 사용한다.
- RAW 사용 시 `standardUnitOrder`는 `999`로 둔다.
- 대응 가능한 표준단원이 있는데도 RAW로 남겨두는 것은 금지한다.

### 5-7. 신규 단원 추가 원칙
- 신규 단원, 세부 단원, 예외 과목이 생기면
  먼저 `# JS아카이브 표준단원키 마스터 테이블.md`를 갱신한 뒤 사용한다.
- 룰북만 먼저 수정하고 마스터 테이블을 비워 두는 방식은 금지한다.

### 5-8. 검수 기준
- `standardCourse`가 교육과정 체계와 일치하는가
- `standardUnitKey`가 마스터 테이블에 존재하는가
- `standardUnit`이 key와 정확히 대응하는가
- `standardUnitOrder`가 key의 순서와 일치하는가
- 대응 단원이 있는데 RAW를 쓰고 있지 않은가
- 2015 / 2022 체계를 혼합 오기하지 않았는가

### 9-4. SVG · table HTML 무결성 규칙 (데이터 작성 기준)

#### SVG 내부 규칙

**[SVG-1] SVG 내부 `<br>` 절대 금지**
`<svg>...</svg>` 내부에는 `<br>` 태그를 단 한 개도 넣지 않는다.
`<line>`, `<path>`, `<text>`, `<circle>`, `<g>` 사이의 줄바꿈은 JS 문자열 `\n`으로만 처리한다.

```text
잘못된 예
"<svg ...><br>  <line .../><br>  <text ...>"

올바른 예
"<svg ...>\n  <line .../>\n  <text ...>"
```

**[SVG-2] SVG 내부 LaTeX 금지**
`<text>` 태그 안에 `$...$`, `\frac`, `\sqrt` 등 LaTeX를 직접 넣지 않는다.
수식 라벨이 필요하면 평문으로만 표기한다. (예: `1/2`, `√3`)
LaTeX가 필요한 수식 라벨은 SVG 밖 HTML 영역에 둔다.

**[SVG-3] 다중 줄 텍스트**
`<text>` 안에서 줄을 나눠야 할 경우 `<br>` 대신 `<tspan>` 사용.

```xml
<text x="50" y="30">
  <tspan x="50" dy="0">윗줄</tspan>
  <tspan x="50" dy="14">아랫줄</tspan>
</text>
```

#### table 내부 규칙

**[TABLE-1] table/tr/td 사이 `<br>` 금지**
`<div>`, `<table>`, `<tr>`, `<td>`, `<th>` 태그 직전·직후에 `<br>`을 넣지 않는다.

```text
잘못된 예
"<div style="text-align:center;"><br><table ...><br><tr>..."

올바른 예
"<div style="text-align:center;"><table ...><tr>..."
```

**[TABLE-2] table 인라인 스타일 남용 금지**
레이아웃 핵심 규칙은 엔진의 전용 selector가 담당한다.
데이터에서 `font-size`, `max-width`, `margin:0 auto` 같은 강한 배치 스타일을 과도하게 인라인으로 넣지 않는다.
정말 필요한 최소한의 셀 보조 스타일만 허용한다.

#### 출력 전 자가 점검

* [ ] SVG 내부 `<br>` 0개인가
* [ ] SVG 내부 `$` 기호 0개인가
* [ ] `<table>` 직후 `<br>` 0개인가
* [ ] 발문 표가 있다면 `.question-table-wrap` 또는 `.question-table` 전략과 일치하는가

### 9-5. 문항 유형 분류 및 렌더링 정책 (Stage E v1.4.3 기준)

Stage E v1.4.3 기준으로 `renderExam`은 문항 유형(`questionType`)과 배치 태그(`layoutTag`)를 분리하여 해석한다.
문항 번호는 원본 data 순서 기준으로 고정되며, 배치 블록이 달라도 번호는 그대로 유지된다.

#### 최상위 원칙

* `wide: true`는 AI가 자동 판정하지 않는다.
* `wide: true`는 사용자가 직접 지정한 경우에만 예외 배치로 처리한다.
* `choices.length === 0`만으로 서술형/특수배치를 자동 판정하지 않는다.
* 주관식 전체를 자동으로 특수배치 대상으로 잡는 행위는 금지한다.
* 특수배치 대상은 반드시 `layoutTag` 또는 `tags`로만 판정한다.
* 시험지 렌더링은 page-wide shared column 누적 방식이 아니라, **block 단위 적층 구조**를 사용한다.

#### 1) 문항 유형 필드 (`questionType`)

문항 성격 분류용 필드:

* `객관식`
* `주관식`
* `서술형`

이 필드는 문항 의미 분류용이며, 직접적으로 페이지 배치 규칙을 결정하지 않는다.

#### 2) 배치 태그 필드 (`layoutTag`)

렌더링 규칙 결정용 필드:

* `grid` : 기본 2열 격자
* `subjective-2up` : 2문항 1블록
* `subjective-4up` : 4문항 1블록
* `fullwidth` : 단독 full-width 예외 배치
* 또는 `wide: true` : 사용자 수동 지정 full-width 예외 배치

즉, 문항 유형(`questionType`)과 배치 태그(`layoutTag`)는 별개로 본다.

#### Group A — 기본 격자 문항

조건:

* `wide !== true`
* `layoutTag`가 없거나 `layoutTag === "grid"`
* 또는 별도 특수배치 태그가 없는 모든 문항

포함:

* 일반 객관식
* 일반 주관식
* 일반 서술형
* 태그 없는 모든 기본 문항

배치:

* qpp 기준 chunk 단위 block 생성
* `qpp=4`면 2x2 블록
* `qpp=6`면 2x3 블록
* 각 chunk는 독립 `grid-container`로 렌더
* page 안에서는 block들을 위에서 아래로 세로 적층
* 높이 기준: `proxyHeight_tight`

#### Group B2 — 태그 기반 2문항 배치

조건:

* `layoutTag === "subjective-2up"`
* 또는 `tags` 배열에 `서술형2`

배치:

* 2문항 1블록
* 좌/우 2열 그리드
* page 안에서 block 단위로 세로 적층
* 공간이 남으면 다음 B2 블록이 아래로 이어짐
* 공간이 부족하면 다음 페이지로 이동
* 높이 기준: `proxyHeight_raw`

#### Group B4 — 태그 기반 4문항 배치

조건:

* `layoutTag === "subjective-4up"`
* 또는 `tags` 배열에 `서술형4`

배치:

* 4문항 1블록
* 좌 2문항 / 우 2문항
* page 안에서 block 단위로 세로 적층
* 높이 기준: `proxyHeight_raw`

#### Group C — 전면 배치 (사용자 지정 예외 문항)

조건:

* `wide: true`
* 또는 `layoutTag === "fullwidth"`

배치:

* 1문항 1페이지 full-width
* 반드시 렌더링 마지막 순서
* 높이 기준: `proxyHeight_raw`

중요:

* `wide: true`는 사용자 수동 지정 전용
* 표/그림/장문 여부를 AI가 자동 판정해서 `wide`를 붙이지 않는다.

#### provisional 상수

```js
const SE_OVERHEAD_COL = 10;
const SE_OVERHEAD_GRID = 20;
const SE_PAGE_TOLERANCE = 5;
const SE_BLOCK_GAP = 15;
```

주의:

* Stage E v1.4.3에서는 shared grid 1페이지 1개 구조를 쓰지 않는다.
* block 간 간격(`SE_BLOCK_GAP`)은 bucket 계산과 실제 렌더 margin-bottom이 일치해야 한다.
* 마지막 block의 gap 처리도 계산과 렌더가 동일해야 한다.
* overflow 검사는 `q-box 자체`, `block 내부 좌/우 column`, `page.body 전체`의 3축으로 본다.
* page.body overflow를 사후 압축으로만 억지 처리하지 말고, bucket 계산과 실제 렌더 높이를 최대한 맞춰야 한다.

### 9-6. 폰트 크기 고정 규칙

엔진 본문 기준(`9.5pt`)에 맞춰 아래 수치를 표준으로 고정한다.
이 수치를 벗어나는 값은 룰북 위반으로 간주한다.

#### 발문 표(question table) 폰트

| 항목                                                 | 고정값         | 근거     |
| -------------------------------------------------- | ----------- | ------ |
| `.question-table` font-size                        | **`8.6pt`** | 엔진 표준값 |
| `.question-table th`, `.question-table td` padding | `3px 5px`   | 엔진 표준값 |
| `.question-table` line-height                      | `1.35`      | 엔진 표준값 |

* 데이터 작성 시 `<table style="font-size:...">` 인라인 폰트 지정은 하지 않는다.
* 발문 표는 `.question-table-wrap table` 또는 `table.question-table` 전용 selector가 자동으로 `8.6pt`를 적용하는 것을 원칙으로 한다.
* 열 수가 많아 특별히 더 작게 해야 할 경우에만 `8pt`까지 허용한다. 그 이하는 금지한다.
* **`.q-box table` 기준 문구는 폐기한다.**

#### SVG 텍스트 폰트

| 용도                              | 고정값         |
| ------------------------------- | ----------- |
| 일반 라벨 (축 문자, 점 이름, 수치)          | **`10px`**  |
| 작은 라벨 (첨자급, 보조 표기)              | **`8px`**   |
| 분수 `<g class="fraction">` 내부 숫자 | **`9px`**   |
| `font-family`                   | **`serif`** |

* `12px`, `14px` 등 큰 폰트는 O, Q, x, y 같은 좌표축 문자에도 사용하지 않는다.
* SVG 내부의 모든 `<text>`는 위 3가지 크기 중 하나만 사용한다.

### 9-7. 표(table) 폭 및 정렬 고정 규칙

#### 새 원칙

1. `max-width 280px / 왼쪽 정렬` 고정 규칙은 **발문 표**에만 강제 적용한다.
2. `.q-box table` 전체 선택자로 시험지/해설지의 모든 table을 일괄 제어하지 않는다.
3. 표 규칙은 반드시 **발문 표 전용 래퍼 또는 클래스** 기준으로 건다.
4. 기타 table(해설 내부 보조표, 별도 UI성 table, 구조용 table)은 발문 표 규칙에 자동 포함시키지 않는다.
5. 즉, 범용 selector가 아니라 명시적 selector로만 제어한다.

#### 권장 방식 A안

```html
<div class="question-table-wrap">
  <table>...</table>
</div>
```

```css
.question-table-wrap { text-align:left; margin:8px 0; }
.question-table-wrap table {
  width:100%;
  max-width:280px;
  margin:0;
  margin-left:0;
  margin-right:auto;
  border-collapse:collapse;
  table-layout:fixed;
  font-size:8.6pt;
  line-height:1.35;
}
```

#### 권장 방식 B안

```html
<table class="question-table">...</table>
```

```css
.question-table {
  width:100%;
  max-width:280px;
  margin:8px 0;
  margin-left:0;
  margin-right:auto;
  border-collapse:collapse;
  table-layout:fixed;
  font-size:8.6pt;
  line-height:1.35;
}
```

#### 금지

* `.q-box table { ... }` 로 전체 table 일괄 제어
* `.q-box table, .q-box th, .q-box td` 식 범용 일괄 제어
* 발문 표와 해설 보조표를 동일 selector로 묶기
* 상위 요소가 다르다는 이유 없이 모든 table에 280px 제한 강제
* `text-align:center`, `margin:0 auto` 등으로 발문 표를 중앙 배치
* 발문 전체를 무조건 표 래퍼로 감싸는 방식

#### 운영 기준

* 발문 바로 아래 배치 원칙은 그대로 유지한다.
* 표는 항상 왼쪽 정렬한다.
* 가운데 정렬 금지.
* 오른쪽 정렬 금지.
* 시험지와 해설지 모두 **발문 표 전용 selector일 때만** 동일 규칙을 적용할 수 있다.
* `renderSol`, 기타 table 구조에 자동 영향 가는 범용 selector는 금지한다.

# 왕지교육 통합 수강·시간표·수납 연동 설계서

> 작성일: 2026-05-15  
> 목적: AP Math, 씨매쓰 초등, EIE 영어학원을 함께 운영할 때 필요한 복수 수강, 통합 시간표, 시간 충돌 감지, 수납 연동 구조를 정리한다.  
> 위치: 왕지교육 홈페이지 이전 전 선행 구현 검토 문서  
> 전제: 홈페이지 이전은 마지막 단계이며, 그 전에 실제 학원 운영에서 필요한 기능을 먼저 설계·구현한다.

---

## 0. 핵심 결론

왕지교육은 AP Math 하나만 운영하는 구조가 아니다.

왕지교육 아래에는 다음 세 교육 브랜드가 함께 있다.

```text
왕지교육
├─ AP Math
│  └─ 중·고등 수학
├─ 씨매쓰 초등
│  └─ 초등 수학·사고력
└─ EIE 영어학원
   └─ 초·중·고 영어
```

따라서 앞으로의 운영 시스템은 다음 원칙을 따라야 한다.

```text
외부 홈페이지:
AP Math / 씨매쓰 초등 / EIE 영어학원 = 3 : 3 : 3 비중

내부 시스템:
공통 운영 기능 + AP Math 심화 기능 + 향후 씨매쓰/EIE 확장 가능 구조

학생 관리:
학생 1명은 여러 브랜드 수업을 동시에 수강할 수 있음

시간표:
데이터는 하나로 관리하고,
화면은 브랜드별 / 통합 / 비교 보기로 나눔

수납:
학생 기준으로 브랜드별 청구를 합산하거나 분리해 관리
```

## 1. 왜 복수 수강 구조가 필요한가

왕지교육에서는 한 학생이 여러 수업을 동시에 수강할 수 있다.

예시:

```text
학생 A
├─ AP Math 수학반
└─ EIE 영어학원 영어반

학생 B
├─ 씨매쓰 초등
└─ EIE 영어학원

학생 C
├─ 씨매쓰 초등
├─ AP Math
└─ EIE 영어학원
```

이런 구조에서는 학생을 특정 브랜드 하나에만 묶으면 안 된다.

잘못된 구조:

```text
student.branch = "apmath"
```

이 방식은 학생이 영어와 수학을 동시에 듣는 상황을 표현하기 어렵다.

올바른 구조:

```text
학생 1명
└─ 여러 개의 수강 등록 enrollment
   ├─ AP Math 수강
   ├─ 씨매쓰 초등 수강
   └─ EIE 영어학원 수강
```

즉, 학생은 하나이고, 수강 정보가 여러 개 붙는 구조가 필요하다.

## 2. 핵심 데이터 구조

### 2-1. 학생

학생은 왕지교육 전체 기준으로 1명만 존재한다.

```text
students
- id
- name
- school_name
- grade
- parent_phone
- student_pin
- status
```

학생이 AP Math와 EIE 영어학원을 동시에 다녀도 학생 레코드는 하나다.

---

### 2-2. 수강 등록

학생과 수업을 연결하는 핵심 구조다.

```sql
CREATE TABLE student_enrollments (
  id TEXT PRIMARY KEY,
  student_id TEXT NOT NULL,
  branch TEXT NOT NULL,
  -- apmath / cmath / eie
  class_id TEXT NOT NULL,
  status TEXT DEFAULT 'active',
  -- active / paused / ended
  start_date TEXT,
  end_date TEXT,
  tuition_amount INTEGER,
  memo TEXT,
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  updated_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

표시명 기준:

```text
apmath = AP Math
cmath  = 씨매쓰 초등
eie    = EIE 영어학원
```

예시:

```text
홍길동
├─ enrollment 1
│  ├─ branch: apmath
│  └─ class: AP Math 고1 내신반
└─ enrollment 2
   ├─ branch: eie
   └─ class: EIE 영어 중등반
```

---

### 2-3. 수업 시간 슬롯

현재 시간표가 문자열 중심이면 충돌 감지가 어렵다.  
정확한 시간표 분석을 위해 수업 시간은 구조화해야 한다.

```sql
CREATE TABLE class_time_slots (
  id TEXT PRIMARY KEY,
  class_id TEXT NOT NULL,
  day_of_week TEXT NOT NULL,
  -- mon / tue / wed / thu / fri / sat / sun
  start_time TEXT NOT NULL,
  -- HH:MM
  end_time TEXT NOT NULL,
  -- HH:MM
  room_name TEXT,
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  updated_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

예시:

```text
AP Math 고1 내신반
- 월 18:00~20:00
- 수 18:00~20:00
- 금 18:00~20:00

EIE 영어 중등반
- 월 19:00~20:30
- 수 19:00~20:30
```

## 3. 통합 시간표 설계

### 3-1. 기본 원칙

시간표는 하나의 데이터로 관리한다.

하지만 화면에서는 여러 방식으로 볼 수 있어야 한다.

```text
전체 시간표
├─ AP Math 보기
├─ 씨매쓰 초등 보기
├─ EIE 영어학원 보기
├─ 통합 보기
├─ 조합 비교 보기
├─ 학생별 보기
├─ 선생님별 보기
└─ 교실별 보기
```

---

### 3-2. 브랜드별 보기

#### AP Math 보기

```text
AP Math 수업만 표시
- 중등 수학
- 고등 수학
- 내신반
- 시험 대비반
- 문제은행/OMR/클리닉 연결 가능
```

#### 씨매쓰 초등 보기

```text
씨매쓰 초등 수업만 표시
- 초등 수학
- 사고력
- 초등 학습 습관 관리
```

#### EIE 영어학원 보기

```text
EIE 영어학원 수업만 표시
- 초등 영어
- 중등 영어
- 고등 영어
- 단어/문법/독해/듣기 관리
```

---

### 3-3. 통합 보기

통합 보기는 원장용 운영 화면이다.

```text
AP Math + 씨매쓰 초등 + EIE 영어학원 전체 시간표를 한 화면에서 확인
```

통합 보기에서 확인할 수 있어야 하는 것:

```text
- 같은 시간대 전체 수업 수
- 교실 사용 현황
- 선생님 수업 배정
- 학생 복수 수강 충돌
- 특정 시간대 과밀
- 특정 요일 과밀
```

---

### 3-4. 조합 비교 보기

필요한 경우 특정 브랜드끼리만 겹쳐서 볼 수 있어야 한다.

예시:

```text
비교 보기:
☑ AP Math
☐ 씨매쓰 초등
☑ EIE 영어학원
```

가능 조합:

```text
AP Math + EIE 영어학원
AP Math + 씨매쓰 초등
씨매쓰 초등 + EIE 영어학원
AP Math + 씨매쓰 초등 + EIE 영어학원
```

용도:

```text
- 수학과 영어를 같이 듣는 학생 충돌 확인
- 초등과 영어 수업 시간 겹침 확인
- 씨매쓰에서 AP Math로 이동하는 학생 시간 확인
- 특정 요일 학원 전체 혼잡도 확인
```

## 5. 충돌 감지 로직

### 5-1. 학생 시간 충돌

같은 학생이 수강 중인 모든 active enrollment를 조회한다.

그 뒤 각 enrollment의 class_time_slots를 비교한다.

충돌 조건:

```text
같은 요일이고,
A.start_time < B.end_time
AND
B.start_time < A.end_time
이면 충돌
```

예시:

```text
AP Math
월 18:00~20:00

EIE 영어학원
월 19:00~20:30

18:00 < 20:30
19:00 < 20:00

=> 충돌
```

---

### 5-2. 선생님 시간 충돌

같은 선생님이 같은 시간에 두 수업을 맡으면 충돌이다.

```text
김선생
월 17:00~18:30 EIE 영어
월 18:00~20:00 AP Math

=> 18:00~18:30 충돌
```

---

### 5-3. 교실 충돌

같은 교실에 같은 시간대 수업이 두 개 이상 배정되면 충돌이다.

```text
1강의실
월 18:00~20:00 AP Math
월 19:00~20:30 EIE 영어

=> 19:00~20:00 교실 충돌
```

---

### 5-4. 연속 수업 / 휴식 부족

완전 충돌은 아니지만 운영상 주의가 필요한 경우다.

```text
김선생
16:00~17:00 씨매쓰
17:00~18:30 EIE

=> 충돌은 아니지만 휴식 시간 없음
```

권장 경고:

```text
⚠ 연속 수업
휴식 시간 확인 필요
```

---

### 5-5. 학생 대기 시간 과다

학생이 같은 날 여러 수업을 듣는데 중간 대기 시간이 너무 길면 표시한다.

예시:

```text
홍길동
16:00~17:00 씨매쓰
20:00~22:00 AP Math

대기 시간: 3시간

⚠ 대기 시간 과다
```

기준 예시:

```text
대기 90분 이상 → 확인 필요
대기 120분 이상 → 조정 권장
```

## 6. 경고 표현 기준

### 6-1. 내부 운영자 화면

원장/선생님 화면에서는 명확하게 표시한다.

```text
⚠ 시간표 충돌
⚠ 수강 시간 조정 필요
⚠ 교실 중복 사용
⚠ 선생님 시간 충돌
⚠ 연속 수업
⚠ 대기 시간 과다
```

강한 표현이 필요한 경우:

```text
운영 충돌
즉시 조정 필요
주의 필요
```

---

### 6-2. 학부모 안내 표현

학부모에게는 불안감을 주지 않게 표현한다.

금지 표현:

```text
위험
문제 발생
충돌
오류
심각
```

권장 표현:

```text
수업 시간이 일부 겹쳐 조정이 필요합니다.
학원에서 확인 후 안내드리겠습니다.

수업 시간 조정이 필요한 부분이 있어 확인 중입니다.
확정 후 다시 안내드리겠습니다.
```

## 8. 수납 연동

### 8-1. 복수 수강 수납

학생이 여러 과목을 수강하면 수납도 학생 기준으로 묶여야 한다.

예시:

```text
홍길동

청구 항목:
- AP Math 5월 수강료        250,000원
- EIE 영어학원 5월 수강료    220,000원
- 복수 수강 할인             -30,000원

총 청구액: 440,000원
납부 상태: 미납
```

---

### 8-2. 필요한 기능

```text
1. 브랜드별 청구
2. 학생별 통합 청구
3. 복수 수강 할인
4. 형제 할인
5. 교재비 별도 청구
6. 특강비 별도 청구
7. 과목별 납부 상태
8. 전체 납부 상태
9. 미납자 알림
10. 영수증 발송
```

---

### 8-3. 수납 구조 원칙

```text
payment = 월별 청구 묶음
payment_items = 세부 항목
billing_adjustments = 할인/감면/추가금
student_enrollments = 수강 정보
billing_templates = 수업별 기본 수강료
```

수납 생성 흐름:

```text
학생의 active enrollment 조회
→ 각 수강 수업의 billing_template 조회
→ payment_items 생성
→ 복수 수강 할인 적용
→ payment 총액 계산
→ 학부모에게 청구 안내
```

## 9. 통합 시간표와 상담 연결

시간표 충돌이 있으면 상담/메모와 연결되어야 한다.

예시:

```text
홍길동 시간표 충돌 감지
→ 학생 카드에 표시
→ 상담 메모 작성
→ 학부모 연락 기록
→ 조정 결과 저장
```

필요 기능:

```text
[상담 메모 작성]
[학부모 안내 기록]
[예외 허용]
[수업 시간 변경]
[조정 완료 처리]
```

## 10. 예외 허용

일부 시간 겹침은 운영상 허용될 수 있다.

예:

```text
수업 앞뒤 5분 겹침
학생이 실제 이동 가능
원장이 확인 후 허용
특정 기간만 임시 허용
```

필요 테이블:

```sql
CREATE TABLE timetable_conflict_overrides (
  id TEXT PRIMARY KEY,
  conflict_type TEXT NOT NULL,
  -- student / teacher / room
  target_id TEXT,
  conflict_key TEXT NOT NULL,
  reason TEXT,
  allowed_by TEXT,
  expires_at TEXT,
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

예외 허용 상태 표시:

```text
⚠ 시간표 충돌
예외 허용됨: 2026-05-31까지
```

## 11. 충돌 로그

충돌 감지 결과는 로그로 남기는 것이 좋다.

```sql
CREATE TABLE timetable_conflict_logs (
  id TEXT PRIMARY KEY,
  conflict_type TEXT NOT NULL,
  -- student / teacher / room / rest / wait
  target_id TEXT,
  branch_pair TEXT,
  class_a_id TEXT,
  class_b_id TEXT,
  day_of_week TEXT,
  overlap_start TEXT,
  overlap_end TEXT,
  severity TEXT DEFAULT 'warning',
  -- info / warning / critical
  status TEXT DEFAULT 'open',
  -- open / ignored / resolved
  resolved_by TEXT,
  resolved_at TEXT,
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

용도:

```text
현재 충돌 목록
해결된 충돌 이력
반복되는 충돌 확인
원장 대시보드 표시
```

## 13. 홈페이지 이전 전 필수 여부

이 기능은 홈페이지 이전 전에 설계에 반드시 포함되어야 한다.

이유:

```text
왕지교육은 세 브랜드를 함께 운영한다.
학생이 여러 과목을 동시에 수강한다.
수납과 시간표는 세 브랜드가 연결되는 핵심 지점이다.
시간표 충돌과 수납 중복을 해결하지 않고 홈페이지만 이전하면 운영이 복잡해진다.
```

따라서 홈페이지 이전 전 선행 구현 로드맵에 반드시 들어가야 한다.

```text
Phase 1 — 공통 운영 기반
- 학생별 복수 수강 구조
- 브랜드별 수강 등록
- class_time_slots
- 통합 시간표
- 시간표 충돌 감지
- 복수 수강 수납 연결
```

## 14. 최종 정리

```text
시간표는 하나로 관리한다.
보기는 브랜드별·통합별·비교별로 나눈다.

학생은 하나다.
수강 등록은 여러 개다.

수납은 학생 기준으로 묶는다.
청구 항목은 브랜드별로 나눈다.

충돌은 자동으로 감지한다.
원장 화면에는 명확하게 표시한다.
학부모에게는 부드럽게 안내한다.
```

최종 핵심 문장:

```text
왕지교육 통합 운영 시스템에서는
AP Math, 씨매쓰 초등, EIE 영어학원의 시간표와 수납을 따로따로 관리하지 않고,
하나의 학생 기준으로 묶어 관리해야 한다.
```

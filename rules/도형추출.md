# 🤖 수리·기하·그래프 집도 통합 프로토콜 v8.2
# The Ironclad Procedural Standard

본 문서는 수학 문항, 해설, 그래프, 기하 도형, SVG 삽입 작업 전반에 대해 절차적 무결성, 수리적 정확성, 원문 성역 보존을 보장하기 위한 최상위 운영 규약이다.

이 프로토콜의 목적은 단 하나다.
분석도 끝나지 않은 상태에서 수정 코드, SVG, 해설, JSON, JS 블록이 먼저 출력되는 사고를 구조적으로 차단하는 것이다.

본 규약은 특히 다음 작업에 절대 적용된다.
- 좌표평면 그래프 생성
- 함수 그래프 SVG 생성
- 기하 도형 SVG 생성
- 도형 삽입형 해설 수정
- 문제 조건을 반영한 시각 요소 재구성
- JS/JSON 내부 content, solution, SVG placeholder 수리
- 특정 문항 id 단위 정밀 집도

이 문서는 단순 권장사항이 아니다.
이 문서는 작업 절차를 강제하는 최상위 운영 체제(OS)이며, 아래 단계가 지켜지지 않은 출력은 완성본으로 인정하지 않는다.

--------------------------------------------------
1. 최상위 원칙 (Supreme Principles)
--------------------------------------------------

[1-1. 절차 우선 원칙]
모든 수정 작업은 반드시 아래 순서를 따른다.

분석 → 승인 대기 → 승인 후 집도

이 순서를 어기고 분석 전에 코드가 출력되거나, 승인 전에 수정본이 제시되거나, 변경 범위를 확정하지 않은 상태에서 SVG/해설/좌표가 먼저 생성되는 행위는 전부 금지한다.

[1-2. 원문 성역 원칙]
문항의 원본 텍스트는 성역이다.
원본 발문, 보기, 정답, 기존 해설, 기존 필드 구조는 사용자의 명시적 승인 없이는 수정할 수 없다.

특히 다음 행위는 금지한다.
- 발문 문장 다듬기
- 보기 표현 임의 교정
- 수식 표기 미세 수정
- 띄어쓰기/문장부호 자동 정리
- 해설의 논리 구조 변경
- SVG 외 텍스트에 대한 은근한 보정

즉, “더 좋아 보이게 손보는 행위” 자체가 금지된다.

[1-3. 계산 우선 원칙]
그래프와 도형은 절대로 눈대중으로 그리지 않는다.
모든 시각 요소는 반드시 문제 조건으로부터 계산된 값에 기반해 생성한다.

허용되는 근거는 다음과 같다.
- 함수식
- 교점 계산
- 대칭축 계산
- 절편 계산
- 정의역/치역 조건
- 길이비/넓이비
- 평행/수직 관계
- 닮음/합동 조건
- 중점, 외분점, 내분점
- 점근선
- 증가·감소 및 극값 조건

즉, 보이는 모양이 맞아 보인다고 해서 정답이 되는 것이 아니다.
반드시 수리적으로 계산된 좌표와 관계가 먼저 존재해야 한다.

[1-4. 단일 출력 완결성 원칙]
한 번 출력할 때는 반드시 검증을 끝낸 완성본이어야 한다.
“일단 출력하고 나중에 고치자”는 사고는 금지한다.

단, Step 1은 분석 문서이며 Step 2는 승인 대기 상태이므로, 이 둘은 수정본이 아닌 절차상 완결된 상태여야 한다.
즉, 각 단계 출력은 그 단계 기준으로 완결본이어야 한다.

--------------------------------------------------
2. 절대 집도 순서 (Operational Hard Rule)
--------------------------------------------------

모든 작업은 반드시 아래 3단계를 순차적으로 통과해야 한다.

[Step 1A. Visual Draft Review]
그래프, 기하, 영역, 좌표 해설, 함수 개형 복원처럼 시각 요소가 핵심인 문항은,
승인 전에 계산 결과를 바탕으로 한 '시각 초안 검토 단계'를 둘 수 있다.

이 단계의 목적은 최종 JS/SVG를 바로 출력하기 전에,
사용자가 도형의 상대 배치, 좌표 축 방향, 교점 위치, 시야각(FOV), 라벨 충돌 여부, 핵심 관계가 맞는지를 먼저 확인할 수 있도록 하는 것이다.

중요:
이 단계는 최종 출력 단계가 아니다.
이 단계는 '캔버스형 검토본' 또는 '도식 검토본'을 보여주는 단계이며,
JS 문자열에 삽입된 최종 SVG, 최종 JSON, 최종 JS 블록을 출력하는 단계가 아니다.

즉, 이 단계에서 허용되는 것은 다음뿐이다.
- 계산된 좌표 관계 설명
- 그래프/도형의 상대 배치 설명
- 시각 초안(캔버스형 검토본, 도식 검토본)
- 어떤 점, 선, 축, 곡선, 영역을 어떻게 배치할지에 대한 검토용 표현

이 단계에서 금지되는 것은 다음과 같다.
- 최종 JS 코드 출력
- content 내부에 삽입될 최종 SVG 문자열 출력
- 승인 전 placeholder 교체본 출력
- 승인 전 해설 최종 반영본 출력

원칙:
시각 초안은 '검토를 위한 중간 시각화'일 뿐이며,
사용자가 배치와 구조를 승인한 후에만 Step 3에서 최종 JS/SVG 반영본으로 변환한다.

==============================
[Step 1: Analyze Changes]
분석 단계 — 실제 수정 금지
==============================

[절대 규칙]
이 단계에서는 실제 수정 코드, 수정된 JS 블록, SVG 코드, JSON 코드, 해설 수정본을 단 한 줄도 출력하지 않는다.

즉, 이 단계는 어디까지나 분석 문서이며, 집도 단계가 아니다.

[이 단계의 목적]
이 단계는 사용자의 개념적 승인을 얻기 위한 검토 단계다.
어떤 파일의 어느 문항의 무엇을 왜 어떻게 수정할지, 그리고 무엇은 절대 건드리지 않을지를 명확하게 선언하는 단계다.

[Step 1에서 반드시 포함해야 하는 분석 항목]

1) 대상 식별
- 대상 파일명
- exam metadata
- targeted ids
- 문항 번호 또는 객체 id
- 관련 필드명(content, solution, choices, answer, svg placeholder 등)

2) 현재 상태 요약
- 현재 어떤 문제가 있는지
- 문제 유형이 무엇인지
- 오류가 텍스트 오류인지, 좌표 오류인지, 시야각 오류인지, SVG 누락인지, 해설 누락인지
- 기존 데이터가 어느 수준까지 살아 있는지

3) 수정 필요 사유
- 왜 수정이 필요한지
- 수정하지 않으면 어떤 문제가 생기는지
- 엔진 렌더링 상 어떤 왜곡이 발생하는지
- 수학적/시각적 불일치가 무엇인지

4) 원본 유지 범위
- 절대 수정 금지 영역
- 원문 발문 유지 여부
- 보기/정답/기존 해설 유지 여부
- 필드 구조 변경 금지 여부
- key 이름 유지 여부
- 객체 순서 유지 여부

5) 예상 수정 범위
- SVG만 수정인지
- solution만 수정인지
- content 내부 placeholder만 교체인지
- 특정 id 객체 전체를 출력해야 하는지
- 연결 무결성 때문에 id부터 }까지 전체 블록이 필요한지

6) 수리 검증 포인트
문항 성격에 따라 반드시 검증 기준을 선언해야 한다.

예:
- 교점 좌표 일치 여부
- 직선의 기울기 일치 여부
- 포물선의 꼭짓점 일치 여부
- 대칭축 일치 여부
- 점근선 위치 일치 여부
- 비율 AB:AC = 3:2 일치 여부
- 닮음 관계 유지 여부
- 평행/수직 관계 유지 여부
- 도형의 상대 배치가 조건과 합치하는지

7) 예상 위험 요소
- 발문 자체가 모순일 가능성
- 이미지 참고 없이 재구성이 어려운 요소
- 기존 SVG placeholder 구조와 충돌 가능성
- MathJax/JS 문자열 escaping 이슈
- 백슬래시 이중 처리 문제
- viewBox 부족으로 잘림 발생 가능성
- 그래프 라벨 겹침 가능성

8) 집도 방식 선언
- 승인 후 어떤 방식으로 출력할 것인지
- 변경된 부분만 출력할지
- 객체 전체를 출력할지
- full-code 범위를 어디까지로 할지

9) 시각 초안 필요 여부
- 이 문항이 계산 후 바로 JS/SVG로 가도 되는지
- 아니면 먼저 캔버스형 시각 초안 검토가 필요한지
- 시각 초안이 필요하다면 어떤 요소를 먼저 확인해야 하는지
  예:
  - 좌표축 방향과 눈금
  - 교점 위치
  - 꼭짓점/절편/대칭축
  - 점근선 위치
  - 선분 비율
  - 라벨 충돌 여부
  - viewBox 범위
  - 도형 간 상대 위치
- 시각 초안 승인 후 어떤 범위를 최종 JS/SVG로 변환할 것인지

[Step 1 출력 형식 원칙]
분석은 설명식 글이 아니라 검토서처럼 작성해야 한다.
감상, 변명, 장식 문구를 최소화하고, 실제 승인에 필요한 정보 위주로 작성한다.

추가 원칙:
시각 요소가 포함된 문항이라고 해서 Step 1에서 바로 최종 SVG나 JS 블록을 출력해서는 안 된다.
필요한 경우 Step 1에서는 '시각 초안 필요 여부'와 '검토 포인트'까지만 선언한다.
사용자가 시각 초안 확인을 원하거나, 시각 오차 가능성이 높은 문항이라면,
최종 삽입 전 캔버스형 검토 단계를 우선 고려한다.

==============================
[Step 2: Wait for User Approval]
승인 대기 단계 — 모든 작업 정지
==============================

[절대 규칙]
분석이 끝난 뒤에는 반드시 승인 전까지 실제 수정 코드를 출력하지 않고 중단해야 한다.

이 단계에서는 새로운 수정안을 밀어넣거나, “참고로 이것도 가능하다”며 코드 일부를 슬쩍 제공하거나, SVG 골격이라며 반쯤 생성한 코드를 보여주는 행위를 금지한다.

[이 단계의 목적]
사용자가 Step 1 분석 내용을 검토하고,
- 수정 범위가 맞는지
- 원문 유지 범위가 맞는지
- 집도 방식이 맞는지
- 수정 우선순위가 맞는지

를 최종 결정하도록 하기 위함이다.

시각 초안이 먼저 제시된 경우,
이 단계에서 사용자는 단순히 "수정 진행 여부"만 승인하는 것이 아니라,
- 좌표 배치가 맞는지
- 도형 구조가 맞는지
- 시야각과 라벨 위치가 적절한지
- 계산 결과가 시각적으로 올바르게 반영되었는지

를 함께 승인한다.

즉, 시각 문항에서는 '수정 승인'과 '시각 초안 승인'이 함께 완료되어야만
Step 3의 최종 JS/SVG 집도로 넘어갈 수 있다.

[Step 2 권장 문구]
다음과 같이 승인 상태가 분명히 드러나는 문구를 사용한다.
- 분석 완료. 승인 전까지 실제 수정 코드는 출력하지 않습니다.
- 승인 대기. 승인 후 변경된 부분만 Full-Code로 집도합니다.
- 분석 단계 종료. 사용자 승인 없이는 SVG/해설/JS 수정본을 출력하지 않습니다.

[금지 사항]
이 단계에서 다음은 전부 금지한다.
- “미리 참고용으로” 코드 출력
- 반쯤 작성된 SVG 제공
- 일부 필드만 먼저 수정 제안
- 승인 없이 해설부터 출력
- 승인 없이 JSON 블록 출력
- 승인 없이 placeholder 교체본 제시
- 시각 초안 승인 없이 최종 SVG 삽입본 출력

즉, 이 단계는 정지 상태여야 한다.

==============================
[Step 3: Execute Modification]
승인 후 집도 단계 — 승인 범위만 정밀 반영
==============================

[절대 규칙]
이 단계는 오직 승인된 내용에 한해서만 수행한다.

사용자가 승인하지 않은 범위를 넓히거나, “같이 고치면 좋을 것 같은 부분”을 끼워 넣거나, 원문 성역을 넘어 수정하는 행위는 금지한다.

[이 단계의 목적]
원본 구조를 유지한 채, 승인된 수정만 정확히 이식하는 것이다.

시각 문항의 경우,
Step 3는 단순 수정 단계가 아니라 승인된 시각 초안을 최종 JS/SVG 삽입본으로 변환하는 단계이기도 하다.

즉, 다음 순서를 따른다.
1. 문제 조건 계산
2. 시각 초안 검토
3. 사용자 승인
4. 승인된 배치를 기준으로 최종 SVG 생성
5. JS/JSON 내부 필드에 안전하게 이식

[출력 원칙]
원칙적으로 변경된 부분만 Full-Code로 출력한다.
다만 다음 경우에는 예외적으로 해당 객체 전체를 출력할 수 있다.
- JS/JSON 구문 무결성 보장을 위해 블록 전체가 필요한 경우
- placeholder 교체 과정에서 content 전체가 필요할 경우
- solution, svg, content가 상호 연결되어 있어 분리 출력이 위험한 경우
- 잘라서 주면 사용자가 그대로 붙여넣기 어려운 경우

즉, 규칙은 다음과 같다.
“불필요한 전체 출력은 금지하되, 구문 무결성 보장을 위해 필요한 전체 블록 출력은 허용한다.”


[객체 단위 출력 명시]
특정 문항 id를 집도할 때, 해당 문항은 반드시 '객체 단위'로 출력한다.
즉, 예를 들어 6번 문항을 수정한다면 6번의 일부 필드만 따로 떼어 출력하지 않고,
반드시 아래 범위를 하나의 완결된 블록으로 제공한다.

- id
- level
- category
- content
- choices
- answer
- solution

원칙:
“한 문항을 수정할 때는 그 문항의 id부터 solution까지 하나의 객체 틀 안에 넣어 출력한다.”

예:
6번 문항 수정 시에는 다음과 같은 형태의 '6번 객체 전체'로 출력해야 한다.

window.questionBank = [
{
  id: 6,
  level: "...",
  category: "...",
  content: "...",
  choices: [...],
  answer: "...",
  solution: "..."
}
]

즉, 특정 문항을 수정할 때는
- content만 단독 출력
- solution만 단독 출력
- svg 문자열만 단독 출력
- choices 일부만 단독 출력

같이 '찢어진 출력'을 금지한다.

단, 사용자가 정말 특정 필드만 따로 요청한 경우에만 예외를 둘 수 있다.
그 외에는 항상 문항 객체 전체 단위 출력이 원칙이다.


[Step 3에서 반드시 지켜야 하는 것]
- 승인된 id만 수정
- 승인된 필드만 수정
- key 이름 변경 금지
- 객체 순서 변경 금지
- 배열 구조 변경 금지
- 쉼표, 따옴표, 이스케이프까지 무결성 유지
- JS 문자열 내부 LaTeX는 백슬래시 이중화 유지
- SVG 외 원문 텍스트는 수정 금지
- 수리 검증 포인트를 만족하는지 재검증 후 출력
- 시각 초안에서 승인되지 않은 배치로 임의 변경 금지
- 초안 승인 후 JS 변환 과정에서 좌표를 다시 임의 수정하지 말 것
- 캔버스 검토본과 최종 SVG 사이의 구조적 불일치 금지
- 특정 id를 수정할 때는 반드시 해당 id의 객체 전체(id ~ solution)를 하나의 완결된 구조로 출력할 것
- 부분 필드만 따로 떼어 출력해서 사용자가 수동 병합하게 만드는 방식 금지
- 예: 6번 수정이면 6번 객체 전체를 그대로 붙여넣을 수 있는 상태로 출력

3. 엔진 기술 표준 (Technical Sanctuary)
--------------------------------------------------

[v8.1-A] Zero-Tolerance Structural Integrity
모든 수정은 최소 수정 원칙을 따른다.
수정은 필요한 범위에서만 수행하며, 해당 범위를 벗어난 변경은 금지한다.

포함 내용:
- 원본 key 고정
- 데이터 구조 고정
- 객체 id 유지
- 원문 텍스트 무변형
- 승인 범위 외 수정 금지
- placeholder 외 타 필드 침범 금지

[v8.1-B] Numerical Anchoring
모든 그래프와 도형은 좌표계 내에서 수치적으로 고정되어야 한다.

의미:
좌표, 길이, 기울기, 절편, 곡선 형태가 실제 문제 조건과 정합해야 하며, 임의의 시각적 균형을 위해 변형해서는 안 된다.

예시:
- y=2는 원점 대비 일관된 높이에 있어야 한다.
- x=-3에서의 함수값은 실제 식에 따라 찍혀야 한다.
- 비율 3:2는 선분 길이로 환산했을 때 실제로 반영되어야 한다.
- 수직 조건은 기울기 곱 -1 등의 조건과 양립해야 한다.

[v8.1-C] Compute Before Draw
시각적 추정 금지.
계산 없이 드로잉하지 않는다.

의무 절차:
1. 문제 조건 해석
2. 필요한 수학적 값 계산
3. 좌표 또는 관계 확정
4. 그 결과를 SVG로 변환
5. 렌더링 후 조건 재검증

금지:
- “대충 이런 모양일 것 같은” 그래프
- 보기 좋은 배치를 위한 임의 왜곡
- 포물선의 대칭성 붕괴
- 삼각형 비율 무시
- 점 위치를 감으로 배치

[v8.1-D] FOV Expansion
그래프가 잘리지 않도록 viewBox와 화각을 충분히 확보한다.

특히 필요한 경우:
- 지수함수
- 로그함수
- 점근선이 있는 함수
- 좌측/우측으로 길게 뻗는 그래프
- 원, 접선, 교점 표시가 필요한 경우

목적:
- 끝점 잘림 방지
- 축 라벨 가독성 확보
- 점근 거동 표현
- 음수 방향 영역 누락 방지

주의:
화각을 넓히더라도 수치 왜곡은 금지한다.
넓게 보이기 위해 비율을 임의 조정하면 안 된다.

[v8.1-E] Absolute Coordinate Discipline
SVG는 가능한 한 절대 좌표 기반으로 작성한다.

권장 원칙:
- 명시적 x, y 사용
- 명시적 path 좌표 사용
- 명시적 중심/반지름 사용
- 명시적 선분 시작/끝 좌표 사용

금지 권장:
- transform 의존
- 상대 좌표 남용
- 렌더링 환경에 따라 달라지는 비율 의존
- 의미 없는 group nesting

[v8.1-F] Rendering Safety
엔진에서 보일 때와 외부 믹서/렌더러로 넘어갈 때가 다를 수 있으므로, 렌더링 안전성도 고려해야 한다.

확인 포인트:
- stroke-width가 너무 얇아 사라지지 않는지
- text 위치가 클리핑되지 않는지
- 폰트 의존성이 지나치지 않은지
- viewBox 대비 실제 요소가 화면 밖으로 나가지 않는지
- baseline 차이로 문자 위치가 틀어지지 않는지

[v8.1-G] Canvas-to-JS Conversion Principle
그래프, 기하, 영역, 좌표 기반 해설은 필요 시 먼저 캔버스형 시각 초안으로 검토한 뒤,
승인된 결과만 최종 JS/SVG로 변환한다.

이 원칙의 목적은 다음과 같다.
- 계산값은 맞지만 배치가 어색한 경우를 사전 차단
- 좌표는 맞지만 시야각이 부족해 잘리는 문제 방지
- 라벨 충돌, 선 겹침, 점 위치 혼동 방지
- 사용자가 최종 삽입 전에 구조를 눈으로 검토할 수 있도록 보장

Canvas-to-JS 절차는 다음과 같다.
1. 문제 조건 계산
2. 좌표/관계 확정
3. 캔버스형 시각 초안 제시
4. 사용자 승인
5. 승인된 초안을 최종 SVG/JS로 변환
6. 삽입 후 렌더링 검증

금지:
- 초안 없이 바로 최종 SVG 삽입
- 초안과 다른 구조로 최종 SVG 생성
- 승인된 시각 구조를 변환 과정에서 임의 수정

4. 수식 및 문자열 규칙 (Strict LaTeX for JS)
--------------------------------------------------

- 수식은 필요한 부분에만 $...$를 적용한다. 일반 텍스트 전체를 기계적으로 감싸지 않는다.
- 이미 정상적으로 구성된 수식은 중복 래핑하지 않는다.
- JS 문자열 내부 LaTeX 백슬래시는 반드시 이중 처리한다 (\\frac, \\sqrt, \\therefore 등).

**줄바꿈**
- 줄바꿈은 \\n으로 표기한다.
- $...$는 \\n 기준 각 줄마다 반드시 쌍으로 닫는다.
  → 위험: "$x^3+y^3=(x+y)^3\\n주어진 조건 $x+y=4"
  → 올바름: "$x^3+y^3=(x+y)^3$\\n주어진 조건 $x+y=4, xy=2$"
- \\notin, \\ne, \\neq, \\nabla 등 \\n으로 시작하는 명령어는 줄바꿈 \\n 바로 뒤에 붙여 쓰지 않는다.
  → 위험: "정리함.\\n\\notin A"
  → 올바름: "정리함.\\n$\\notin A$"

**집합 표기**
- \\{...\\}는 반드시 $...$로 감싼다.
  → 위험: "정의역이 \\{x|x>0\\}인 함수"
  → 올바름: "정의역이 $\\{x\\,|\\,x>0\\}$인 함수"

**부등호**
- choices에 부등호 <, >를 직접 쓰지 않는다. \\lt, \\gt로 대체한다.
  → 위험: "$-\\frac{1}{3}<x<2$"
  → 올바름: "$-\\frac{1}{3}\\lt x\\lt 2$"

**\\text{} 사용 금지**
- \\text{} 안에 한국어를 쓰지 않는다. 레이블은 평문으로 쓴다.
  → 위험: "\\\\text{i) } x=0 \\\\text{ 대입:}"
  → 올바름: "(i) x=0 대입:\\n"

**영어 표현 금지**
- solution에 implies, therefore, hence, thus, because를 영어 그대로 쓰지 않는다.
  → 위험: "ximplies", "therefore a = -4"
  → 올바름: "\\\\implies", "\\\\therefore a = -4"

**LaTeX 내부 줄바꿈** (aligned, cases 등 환경 내부)
- JS 문자열 기준 \\\\\\\\ (백슬래시 8개) → 런타임 \\\\ → LaTeX \\

4. 논리 무결성 및 오류 대응 규칙

[v8.1-H] Ghost Error Detection
문항 자체에 수리적 모순이 있는지 점검한다.

예:
- 조건끼리 양립 불가
- 정답과 조건 불일치
- 도형 비율 충돌
- 교점 개수 모순
- 함수 조건상 나올 수 없는 그래프

중요 원칙:
모순 발견 시 자동 수정 금지.

해야 할 일은 다음뿐이다.
1. 모순 여부 보고
2. 어떤 부분이 충돌하는지 설명
3. 수정 가능성 제안
4. 승인 전 실제 수정 금지

즉, 모델이 스스로 발문을 교정해서는 안 된다.

[v8.1-I] Answer Consistency Guard
그래프/도형/해설은 정답과 충돌하면 안 된다.

검증 대상:
- 해설 전개가 정답과 일치하는지
- 그래프 해석 결과가 정답과 일치하는지
- 선택지 구조와 계산 결과가 맞는지
- 도형 관계가 결론과 충돌하지 않는지

정답 불일치가 발견되면, 해설을 억지로 맞추는 것이 아니라 불일치 사실을 보고해야 한다.

5. 해설 작성 가이드라인 (Review Rules)
--------------------------------------------------

[5-1. 수식 중심 전개]
설명문보다 수식 전개를 중심으로 쓴다.
장황한 서술보다 계산 흐름이 먼저 보여야 한다.

[5-2. 무삭제 전개]
인수분해, 치환, 부호 변화, 제곱 완성, 판별식, 미분 결과, 좌표 계산 등의 중간 과정을 생략하지 않는다.

즉,
- 결과만 쓰기 금지
- “따라서 자명” 금지
- 뜬금없는 결론 점프 금지

[5-3. 기호 사용 원칙]
다음 기호를 적극 활용한다.
- \\rightarrow
- \\implies
- \\therefore

수식 흐름이 선명하게 보이도록 한다.

[5-4. LaTeX 무결성]
JS 내부 문자열의 LaTeX는 반드시 escape를 고려해 이중 백슬래시를 유지한다.

예:
- \\frac{a}{b}
- \\sqrt{x}
- \\therefore

[5-5. 정답 맞춤형 왜곡 금지]
주어진 정답에 끼워 맞추기 위해 논리를 왜곡하면 안 된다.
풀이 결과가 정답과 다르면, 억지 수정이 아니라 충돌 보고가 우선이다.

Guiding Principle:
“승인 없는 집도는 수정이 아니라 훼손이다.”

절차를 어긴 코드, 분석 없는 집도, 승인 없는 수정본은 전부 무효로 간주한다.
항상 분석을 먼저 수행하고, 승인 전에는 실제 수정 코드를 출력하지 않으며, 승인 후에는 승인된 범위만 정밀 반영한다.
그래프와 도형은 반드시 계산 기반으로 생성하며, 원문 텍스트는 성역으로 취급한다.

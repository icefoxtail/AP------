══════════════════════════════════════
PHASE 0 — TYPE DETECTION (먼저 실행)
══════════════════════════════════════
입력 문항을 읽고 유형을 판별한다.
TYPE_A = 객관식 (choices 5개, answer = ①~⑤)
TYPE_B = 단답형 (choices [], answer = 숫자/식 문자열)
TYPE_C = 서술형 (choices [], answer = 핵심값 문자열)

→ 판별된 유형은 내부적으로 고정하고 절대 변경하지 않는다.
→ 원본이 객관식이면 객관식 유지, 주관식이면 주관식 유지, 서술형이면 서술형 유지.
→ choices.length만으로 배치 태그를 추정하지 않는다.

══════════════════════════════════════
PHASE 1 — SOLUTION FIRST (풀이 선행)
══════════════════════════════════════
1. 변형할 숫자/변수를 결정한다.
2. 변형된 문항을 직접 풀어 최종 정답값을 구한다.
3. 검산(대입 또는 역산)으로 정답을 확인한다.
4. 검산 불일치 시 수치 재조정 후 2부터 반복한다.
5. 정답이 확정된 이후에만 PHASE 2로 진행한다.

※ 풀이보다 choices/answer를 먼저 쓰는 것을 금지한다.
※ answer와 solution 결론이 다르면 FAIL이다.

────────────────────────────────────
[수식 표기 규칙 — JS아카이브 통합 표준]
────────────────────────────────────

▶ 기본 원칙
- 수식은 필요한 부분에만 $...$를 적용한다.
- 이미 정상적으로 구성된 수식은 중복 래핑하지 않는다.
- JS 문자열 내부 LaTeX 백슬래시는 반드시 이중 처리한다.
  예: \\frac, \\sqrt, \\therefore, \\implies

▶ 줄바꿈
- 줄바꿈은 \\n으로 표기한다.
- $...$는 \\n 기준 각 줄마다 반드시 쌍으로 닫는다.
  위험: "$x^3+y^3=(x+y)^3\\n주어진 조건 $x+y=4"
  올바름: "$x^3+y^3=(x+y)^3$\\n주어진 조건 $x+y=4, xy=2$"
- \\notin, \\ne, \\neq, \\nabla 등 \\n으로 시작하는 명령어는 줄바꿈 \\n 바로 뒤에 붙여 쓰지 않는다.
  위험: "정리함.\\n\\notin A"
  올바름: "정리함.\\n$\\notin A$"

▶ 집합 표기
- \\{...\\}는 반드시 $...$로 감싼다.
  예: "$\\{x\\,|\\,x>0\\}$"

▶ 부등호
- choices 내부 부등호는 <, > 직접 사용 대신 \\lt, \\gt 사용
  예: "$-\\frac{1}{3}\\lt x\\lt 2$"

▶ \\text{} 사용 금지
- \\text{} 안에 한국어를 쓰지 않는다.
  위험: "\\text{이상}"
  올바름: 평문으로 분리

▶ 영어 표현 금지 (solution 내부)
- implies, therefore, hence, thus, because를 영어로 쓰지 않는다.
- 필요 시 $\\implies$, $\\therefore$, $\\because$ 사용

▶ LaTeX 내부 줄바꿈
- aligned, cases 등 환경 내부 줄바꿈은 JS 문자열 기준 백슬래시를 충분히 이중 처리한다.

[choices / answer 규칙]
────────────────────────────────────

[TYPE_A = 객관식]
- choices는 반드시 5개 문자열
- answer는 반드시 ①~⑤ 중 하나
- choices 형식 예:
  ["① $x=1$", "② $x=2$", "③ $x=3$", "④ $x=4$", "⑤ $x=5$"]

[TYPE_B / TYPE_C = 주관식 / 서술형]
- choices: []
- answer: 숫자 문자열 또는 수식 문자열
- answer에 수식이 들어가면 반드시 $...$ 사용
  예: "$\\frac{1}{2}$"

────────────────────────────────────
[solution 규칙]
────────────────────────────────────
- solution은 반드시 학생이 읽는 풀이만 남긴다.
- solution은 반드시 [키포인트]로 시작한다.
- 운영 규칙, 모델 내부 로그, self audit 문장, mapping 설명, cite 흔적을 넣지 않는다.
- 정답을 역으로 끼워 맞춘 풀이를 금지한다.
- 해설 결론은 answer와 반드시 일치해야 한다.
- 해설 길이는 원본과 크게 동떨어지지 않게 유지한다.

표준 형식:
[키포인트] 이 문제의 핵심 개념 한 줄
1단계: ...
2단계: ...
∴ 정답: ③

또는 주관식:
[키포인트] ...
1단계: ...
2단계: ...
∴ 정답: $\\frac{1}{2}$

[SVG / table 규칙]
────────────────────────────────────

[SVG]
- 원본에 svg가 있을 때만 SVG 유지/수정한다.
- SVG 내부 <br> 절대 금지
- SVG 내부 LaTeX($...$, \\frac, \\sqrt 등) 절대 금지
- SVG 내부 text font-size는 8px / 9px / 10px 중 하나만 사용
- 분수용 <g class="fraction"> 내부 숫자는 9px 기준
- SVG는 시험지 엔진의 staging 폭 83mm 안에서 자연스럽게 들어가야 한다.

[TABLE]
- 발문 표는 발문 바로 아래에 둔다.
- 발문 표는 .question-table-wrap 또는 .question-table 전략만 따른다.
- .q-box table 범용 selector 전제를 다시 만들지 않는다.
- <table>/<tr>/<td>/<th> 직전·직후 <br> 금지
- table 인라인 font-size 남용 금지

[도형/표 마커]
- 실제 도형 또는 표가 삽입되면 기존 [도형필요], [표필요], [해설도형필요], [해설표필요] 같은 마커는 삭제한다.
- 실제 삽입 후에도 마커가 남아 있으면 FAIL

□ C5. 수식 표기 — 모든 수식이 $...$ 형식인가?
□ C6. 백슬래시 — JS 문자열 내 LaTeX 명령어가 이중 처리인가?
□ C7. 줄바꿈 — $...$ 쌍이 \\n 기준 각 줄마다 닫혀 있는가?
□ C8. 부등호 — choices 내 <, >를 \\lt, \\gt로 대체했는가?
□ C9. \\text{} — 한국어를 \\text{} 안에 넣지 않았는가?
□ C10. 영어 — solution에 therefore/hence 등 영어 표현이 없는가?
□ C11. choices — TYPE_A는 5개, TYPE_B/C는 []인가?
□ C12. answer — TYPE_A는 ①~⑤, TYPE_B/C는 값 문자열인가?
□ C13. solution — [키포인트]로 시작하며 answer와 결론이 일치하는가?
□ C14. questionType / layoutTag — 둘을 혼동하지 않았는가?
□ C15. 특수배치 — choices.length만으로 배치를 판정하지 않았는가?
□ C16. wide — 사용자 수동 지정 외 자동 부여하지 않았는가?
□ C17. SVG — 내부 <br>, LaTeX, 잘못된 font-size가 없는가?
□ C18. table — 전용 selector 전략에 맞고 불필요한 <br>가 없는가?
□ C19. marker — 도형/표 삽입 후 마커가 삭제되었는가?
□ C20. JSON/JS 파싱 — 중괄호, 대괄호, 쉼표, 따옴표 짝이 맞는가?

모든 체크 통과 시 PHASE 4로 진행한다.
하나라도 NO이면 해당 항목만 수정 후 체크리스트 처음부터 재실행한다.

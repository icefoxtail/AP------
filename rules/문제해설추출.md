# Statement-Solution One-Pass JS Output Protocol
# (Engine-Compatible, Flexible Recovery, Full StandardUnit, Strict Math/Table Rules, Marker-Only Diagram Policy) 

--------------------------------------------------
0. 핵심 원칙
-------------------------------------------------- 

- 목표: 발문(content), 보기(choices), 정답(answer), 해설(solution)을 한 번에 JS 객체로 출력한다.
- 이 공정은 1차 생산용이다.
- 완벽 복제 집착보다 **최대한 복원 + 엔진 호환성 유지 + 후속 검수 가능성 확보**를 우선한다.
- 출력은 반드시 **JS아카이브 엔진에서 바로 동작하는 완성본**이어야 한다.
- 도형/그래프는 이 공정에서 생성하지 않는다. **마커만 표기**한다. 

[최우선 우선순위]
- 1순위: 해설 원칙
- 2순위: 충돌 시 최소 보정 가능

--------------------------------------------------
1. 출력 목표
-------------------------------------------------- 

다음 4개를 반드시 생성한다. 

1. content
2. choices
3. answer
4. solution 

다음 메타데이터도 함께 포함한다. 

- id
- level
- category
- originalCategory
- standardCourse
- standardUnitKey
- standardUnit
- standardUnitOrder 

--------------------------------------------------
2. 출력 형식
-------------------------------------------------- 

반드시 아래 구조를 따른다. 

window.examTitle = "파일명"; 

window.questionBank = [
  {
    id: 1,
    level: "중",
    category: "",
    originalCategory: "",
    standardCourse: "",
    standardUnitKey: "",
    standardUnit: "",
    standardUnitOrder: 0,
    content: "",
    choices: [],
    answer: "",
    solution: ""
  }
]; 

- 필드명 변경 금지
- 필드 추가 금지
- 필드 삭제 금지
- 최종 출력은 code block 하나로만 출력
- code block 바깥 설명 금지 
- 서술형 : layoutTag: "subjective-2up" 적용은 마스터의 지시에 따른다. 절대 임의로 정하지 않는다.
--------------------------------------------------
3. 발문/보기 규칙
-------------------------------------------------- 
- 자료의 학생 풀이 및 정답은 절대 참고하지 않는다.
- 발문과 보기는 원문을 최대한 유지한다.
- 사소한 OCR 오류는 문맥 기반 최소 복원 허용
- 의미가 달라질 수 있는 추정 금지
- 일부만 불명확하면 해당 부분만 "[판독불가]" 사용
- 문장 순서 임의 변경 금지
- 보기 표현 임의 교정 금지
- 보기가 객관식이면 배열 유지, 주관식이면 [] 유지 

--------------------------------------------------
4. 도형/그래프 처리 규칙
-------------------------------------------------- 

- 이 공정에서는 도형/그래프/SVG를 생성하지 않는다.
- 좌표 계산, SVG 생성, 시각 요소 복원은 별도 공정으로 넘긴다.
- 이 공정에서 허용되는 것은 **마커 표기만**이다. 

허용 마커:
- [도형필요]
- [그래프필요]
- [도형 조건 확인 필요]
- [그래프 조건 확인 필요] 

마커 규칙:
- content의 맨 앞 또는 맨 끝에만 넣는다.
- 문장 중간 삽입 금지
- 원문에 없는 도형 설명을 길게 덧붙이지 않는다.
- 도형이 없어도 풀이 가능한 경우는 마커 없이 진행 가능
- 도형이 있어야 정답 확정이 가능한 경우 answer는 "[정답불가]" 사용 가능
- solution에는 필요 시 "[도형 검토 후 보완 필요]"를 넣을 수 있다. 

--------------------------------------------------
5. 정답 규칙
-------------------------------------------------- 

- 객관식: "①" ~ "⑤" 등 원표기 사용
- 주관식: 값 문자열 사용
- 주관식 답에 수식이 포함되면 반드시 $...$ 사용
- 불확실하면 "[정답불가]"
- 사소한 OCR 흔들림만으로 쉽게 "[정답불가]" 처리하지 않는다. 

--------------------------------------------------
6. 해설 규칙
-------------------------------------------------- 

해설은 반드시 아래 구조를 따른다. 

[키포인트]
조건 정리
풀이 과정
결론 

세부 규칙:
- 반드시 직접 풀이한다.
- 정답 끼워맞추기 금지
- 계산 없이 결론만 제시 금지
- 중간 과정 생략 최소화
- 해설의 최종 결과는 answer와 반드시 일치해야 한다.
- answer와 충돌하는 결론 금지
- 결론 문장은 아래 중 하나를 사용한다. 

1) 따라서 정답은 ③이다.
2) 따라서 구하는 값은 $...$이다. 

- 해설에는 운영 규칙, 검수 규칙, 출제 의도, 자기평가, 검산 보고를 넣지 않는다.
- 해설에는 OCR 상태, 원문 이상 가능성, 보기 오류 가능성, 원본 확인 필요, 참고 메모를 넣지 않는다.
- 해설은 학생이 실제로 읽는 최종본만 작성한다.
- 내부 판단 과정이나 검토 메모를 해설 본문에 노출하지 않는다.

--------------------------------------------------
6-1. 난이도 "상" 해설 강화 규칙 (FIRST-LEARNER FULL EXPLANATION STANDARD)
--------------------------------------------------

적용 대상:
- level === "상"

목표:
- 문제를 처음 보는 학생도 해설만 읽고 풀이 흐름을 재현할 수 있어야 한다.
- 단순 정답 확인용 해설이 아니라, 학습용 해설로 작성한다.
- "왜 이 식을 세우는지", "왜 이 방법을 쓰는지", "왜 다음 단계로 넘어가는지"가 드러나야 한다.

[최상위 원칙]

- 상 난이도 해설은 서술형 채점 기준 수준으로 작성한다.
- 중간 논리 점프를 허용하지 않는다.
- 답만 맞는 축약형 풀이를 금지한다.
- 처음 배우는 학생이 막힐 수 있는 지점을 해설 안에서 직접 풀어준다.
- 해설을 읽은 학생이 같은 유형 문제를 다시 풀 수 있어야 한다.

[필수 구조]

해설은 반드시 아래 순서를 따른다.

1. [키포인트]
2. 조건 정리
3. 풀이 방향 설정
4. 풀이 과정
5. 결론

--------------------------------------------------
(1) [키포인트]
--------------------------------------------------

- 이 문제의 핵심 개념 또는 출제 포인트를 1~2문장으로 먼저 밝힌다.
- 학생이 "이 문제가 무엇을 묻는 문제인지" 바로 알 수 있어야 한다.
- 단원명만 적는 식의 형식적 문구는 금지한다.

좋은 예:
- [키포인트] 주어진 식을 바로 계산하기보다, 같은 문자로 치환하여 식의 구조를 단순하게 보는 것이 핵심이다.
- [키포인트] 이 문제는 두 근의 합과 곱을 먼저 구한 뒤, 구하려는 식을 그 값으로 바꾸는 것이 핵심이다.

나쁜 예:
- [키포인트] 이차방정식 문제이다.
- [키포인트] 열심히 계산한다.

--------------------------------------------------
(2) 조건 정리
--------------------------------------------------

- 문제에서 주어진 조건을 수식 중심으로 다시 정리한다.
- 필요한 경우 문자 정의, 범위 조건, 숨은 조건을 함께 적는다.
- 조건이 많으면 항목별로 분리한다.
- 그림이 필요한 문제인데 그림 없이 완전 판단이 어려우면 마커 또는 보완 문구를 사용한다.

필수 원칙:
- 주어진 조건을 해설 초반에 다시 명확히 드러낸다.
- 학생이 문제 본문으로 다시 올라가지 않아도 되게 한다.

예:
- $a+b=5$, $ab=6$
- $x>0$
- $f(1)=3$, $f'(1)=2$
- 점 $A$는 원 위의 점이다.

--------------------------------------------------
(3) 풀이 방향 설정
--------------------------------------------------

- 바로 계산에 들어가지 말고, 왜 이 방법으로 풀지 먼저 1~2문장으로 제시한다.
- 이 단계는 상 난이도 해설에서 매우 중요하다.
- 학생이 발상을 배우는 구간이므로 생략 금지.

예:
- 구하려는 식이 직접 계산되지 않으므로, 먼저 주어진 조건을 이용해 기본값을 만들고 이를 대입하는 방식으로 접근한다.
- 식이 복잡하므로 $t$로 치환하여 이차방정식 형태로 바꾸면 해결할 수 있다.
- 경우가 여러 개로 나뉘므로 먼저 기준을 정해 경우를 분리한다.

금지:
- 아무 안내 없이 바로 식 전개 시작
- "계산하면 된다" 같은 무의미한 방향 제시

--------------------------------------------------
(4) 풀이 과정
--------------------------------------------------

- 반드시 단계별로 전개한다.
- 각 단계는 앞 단계와 자연스럽게 연결되어야 한다.
- 식 변형 이유가 보여야 한다.
- 전개, 인수분해, 약분, 치환, 정리 과정은 최소 1단계 이상 드러낸다.
- "정리하면", "계산하면", "쉽게 알 수 있다"만으로 중간 과정을 덮지 않는다.

세부 규칙:

[4-1. 한 단계씩 전개]
- 한 줄에 결과만 던지지 않는다.
- 필요한 경우 1단계, 2단계, 3단계처럼 나누어도 된다.

[4-2. 변형 이유를 드러낼 것]
- 왜 인수분해하는지
- 왜 치환하는지
- 왜 경우를 나누는지
- 왜 공식 적용이 가능한지
를 짧게라도 보여준다.

[4-3. 계산 생략 최소화]
특히 아래는 생략 금지:
- 다항식 전개
- 인수분해 핵심 단계
- 근의 공식 대입 후 정리 핵심 단계
- 절댓값/부등식의 경우 나누기
- 로그/지수의 동치 변형
- 삼각함수 값 대입의 근거
- 수열 일반항/합 공식 적용 과정
- 미분 후 부호 해석 핵심 단계
- 적분식 세팅과 구간 해석

[4-4. 경우 나누기]
- 경우를 나누는 기준을 먼저 밝힌다.
- 경우1, 경우2를 명확히 구분한다.
- 각 경우의 결론을 따로 적고 마지막에 통합한다.

[4-5. 치환]
- 치환은 반드시 선언 후 사용한다.
- 치환한 이유가 구조 단순화인지 함께 보여준다.

예:
- 식에 $x^2$와 $x^4$가 함께 있으므로 $t=x^2$로 두면 이차방정식 형태가 된다.
- $t=x^2$로 두면
- $t \ge 0$ 임도 함께 확인한다.

[4-6. 공식 사용]
- 공식을 그냥 던지지 않는다.
- 어떤 공식인지 또는 왜 적용 가능한지 짧게 밝힌다.

예:
- 두 근의 합과 곱을 이용하면
- 사인법칙을 적용하면
- 등비수열의 합 공식을 적용하면

[4-7. 수식 표현]
- 논리 전개가 분명한 부분에서는 \\implies 사용을 권장한다.
- 결론 직전 정리에는 \\therefore 사용 가능하다.
- 여러 줄 식 정리가 필요한 경우 \\begin{aligned} ... \\end{aligned} 사용을 허용한다.
- 단, 기호를 장식처럼 남용하지 않는다.

--------------------------------------------------
(5) 결론
--------------------------------------------------

- 결론은 answer와 반드시 일치해야 한다.
- 아래 형식 중 하나를 사용한다.

1) 따라서 정답은 ③이다.
2) 따라서 구하는 값은 $...$이다.

- 객관식인데 숫자만 쓰고 끝내지 않는다.
- 주관식인데 말만 하고 수식을 안 적는 것도 금지한다.

--------------------------------------------------
[학생 이해 중심 추가 원칙]
--------------------------------------------------

- 해설은 "출제자 관점"이 아니라 "학생 학습 관점"으로 작성한다.
- 처음 보는 학생이 막힐 만한 포인트를 해설 안에서 먼저 풀어준다.
- 한 줄 요약형 풀이보다, 재현 가능한 풀이를 우선한다.
- 어려운 식일수록 "무엇을 먼저 할지"를 먼저 말해 준다.
- 정답 도출보다 풀이 재현성을 우선한다.

반드시 챙길 것:
- 왜 이 방법을 쓰는가
- 어디서 식이 바뀌는가
- 어떤 조건 때문에 해를 버리는가
- 최종적으로 무엇이 답인가

--------------------------------------------------
[권장 서술 톤]
--------------------------------------------------

- 설명은 짧고 분명하게 한다.
- 불필요한 감탄, 장황한 잡설, 운영자식 메타 설명은 넣지 않는다.
- 한국어 설명 + 수식 전개가 자연스럽게 이어지게 작성한다.
- 일반 텍스트를 억지로 전부 수식화하지 않는다.

좋은 예:
- 먼저 구하려는 식을 바로 계산할 수 없으므로, 주어진 조건으로 바꿔 쓸 수 있는 형태를 찾는다.
- 이 식은 $x^2$에 대한 이차식으로 볼 수 있으므로 치환한다.
- 조건 $x>0$을 만족하는 해만 남긴다.

나쁜 예:
- 자명하다.
- 당연하다.
- 쉽게 알 수 있다.
- 계산하면 나온다.

--------------------------------------------------
[금지 사항]
--------------------------------------------------

- 결과만 맞는 축약형 풀이
- 정답을 기준으로 역으로 해설 구성
- 핵심 전개를 "정리하면" 한마디로 생략
- 경우 나누기 생략
- 학생이 이해할 수 없는 과도한 압축
- 본문보다 더 어려운 표현으로 해설 작성
- 상 난이도라는 이유로 불친절한 해설 작성

--------------------------------------------------
[상 난이도 해설 최소 품질 기준]
--------------------------------------------------

아래 중 하나라도 만족하지 못하면 해설 품질 미달로 본다.

- 풀이 방향이 제시되어 있는가
- 중간 계산이 드러나는가
- 결론이 명확한가
- 처음 보는 학생도 흐름을 따라갈 수 있는가

--------------------------------------------------
6-2. 학생용 해설 금지 규칙
--------------------------------------------------

- 학생용 해설에는 아래 요소를 절대 넣지 않는다.
  - [검산]
  - [검산 또는 조건 대조]
  - [참고]
  - [주의]
  - [보충]
  - [메모]
  - OCR 관련 문장
  - 원문 오류 언급
  - 검토 메모
  - 운영자 요청 문장
  - "정답이 다를 수 있음"
  - "원문 확인 필요"
  - "OCR 확인 필요"
  - "보기 오류 가능성"
  - "의도되었을 수 있음"
  - "[정답불가] 사유 설명"

- 학생용 해설은 완결된 최종 설명만 남긴다.
- 조건 확인이 꼭 필요할 때는 풀이 과정 안에서 자연스럽게 처리하고, 별도 검산 문단으로 분리하지 않는다.

--------------------------------------------------
6-3. 충돌 문항 최소 보정 규칙 (CONFLICT MINIMAL CORRECTION RULE)
--------------------------------------------------

적용 대상:
- 계산 결과와 보기/정답이 논리적으로 일치하지 않는 경우
- OCR 오류, 보기 오탈자, 숫자 누락 등으로 정답이 보기 내에 존재하지 않는 경우

[최상위 원칙]

- 기본 원칙은 원문 유지이다.
- 보정은 예외 상황에서만 허용한다.
- 보정은 “정답을 맞추기 위한 임의 수정”이 아니라 “오류 복원”이다.

--------------------------------------------------
[적용 조건]
--------------------------------------------------

아래 조건을 모두 만족할 때만 보정을 허용한다.

1. 풀이를 통해 계산 결과가 단일 값으로 명확히 결정되는 경우
2. 문제 조건 자체는 논리적으로 정상인 경우
3. 계산 결과가 보기 중 어디에도 존재하지 않는 경우
4. 보기 중 일부가 OCR 오류 또는 단순 오탈자로 판단되는 경우

※ 하나라도 만족하지 못하면 보정 금지 → [정답불가] 처리

--------------------------------------------------
[보정 방식]
--------------------------------------------------

- answer는 계산 결과 기준으로 확정한다.

- choices는 아래 규칙으로 최소 수정한다.

  1) 가장 가능성이 높은 오탈자 보기 1개만 수정
     (숫자 오류, 부호 오류, 자리수 오류 등)

  또는

  2) 기존 보기 하나를 계산 결과 값으로 교체

- 보기 개수(5지선다)는 유지한다.
- 보기 순서 전체 재배열 금지
- 2개 이상 동시 수정 금지
- 문제(content)는 절대 수정하지 않는다.

--------------------------------------------------
[판단 기준]
--------------------------------------------------

다음과 같은 경우에만 보정 수행:

- 28 → 8 (자리수 오류)
- 230 → 30 (OCR 결합 오류)
- -28 → 28 (부호 오류)
- 19 ↔ 1, 9 분리/결합 오류

다음과 같은 경우는 보정 금지:

- 여러 보기 모두 틀린 경우
- 계산 결과가 애매한 경우
- 조건 자체가 의심되는 경우
- 출제 의도를 추정해야 하는 경우

→ 이 경우는 반드시 answer를 "[정답불가]"로 처리

--------------------------------------------------
[해설 규칙]
--------------------------------------------------

- 해설에는 보정 사실을 직접 언급하지 않는다.
- 학생용 해설은 정상 풀이만 제시한다.
- 내부 보정 판단, OCR 오류 추정, 원문 의심 문장은 출력하지 않는다.

--------------------------------------------------
[금지 사항]
--------------------------------------------------

- 문제(content) 수정 금지
- 보기 전체 재작성 금지
- 정답을 맞추기 위한 임의 숫자 생성 금지
- 복수 보기 동시 수정 금지
- 논리적 근거 없이 값 변경 금지

--------------------------------------------------
[우선순위 규칙]
--------------------------------------------------

1순위: 해설 원칙 유지
2순위: 충돌 문항 → 최소 보정 적용
3순위: 보정 불가 → answer를 "[정답불가]" 처리

8. 수식 및 문자열 규칙 (상세)
-------------------------------------------------- 

[8-1. 기본 수식 규칙]
- 수식은 필요한 부분에만 $...$를 사용한다.
- 일반 텍스트 전체를 기계적으로 수식 처리하지 않는다.
- 이미 정상적인 수식은 중복 래핑하지 않는다. 

[8-2. JS 문자열 규칙]
- JS 문자열 내부 LaTeX 백슬래시는 반드시 이중 처리한다.
- 예:
  - \\frac
  - \\sqrt
  - \\implies
  - \\therefore
  - \\because 

[8-3. 줄바꿈 규칙]
- 줄바꿈은 반드시 \\n 사용
- 실제 개행문자를 그대로 넣지 않는다.
- 수식은 줄 단위로 닫혀 있어야 한다.
- 잘못된 예: "$a\\nb$"
- 올바른 예: "$a$\\n$b$" 

[8-4. 명령어 충돌 방지]
- \\notin, \\ne, \\neq, \\nabla 같은 명령은 줄바꿈 \\n과 붙어서 깨지지 않게 한다. 

[8-5. 부등호 규칙]
- choices 내부에서 <, > 직접 사용 금지
- 반드시 \\lt, \\gt 사용 

[8-6. 집합 규칙]
- 집합 표기 \\{...\\}는 반드시 $...$ 안에서 사용
- 예: "$\\{x \\mid x>0\\}$" 

[8-7. 한국어와 \\text{} 규칙]
- \\text{} 안에 한국어 사용 금지
- 한국어는 평문으로 작성한다. 

[8-8. 조합/순열 표기]
- ${}_{n}C_{r}$, ${}_{n}P_{r}$, ${}_{n}H_{r}$ 형태 사용
- 반드시 $...$ 안에서 사용 

[8-9. 해설 기호 사용]
- 다음 기호는 적극 사용 가능:
  - \\rightarrow
  - \\implies
  - \\therefore
- 단, 영어 therefore, implies 같은 평문 사용은 지양한다. 

[8-10. 주관식 정답 규칙]
- 주관식 answer에 수식이 포함되면 반드시 $...$ 사용
- 숫자만이면 plain string 가능
- 여러 값이면 쉼표로 구분하되, 수식값은 각각 $...$로 감싼다. 

[8-11. 문자열 안정성]
- content와 solution은 JS 문자열로 바로 들어가도 깨지지 않아야 한다.
- 따옴표, 백슬래시, 줄바꿈, 괄호 닫힘, 수식 닫힘을 반드시 확인한다.

9. 표(Table) 규칙 (상세)
-------------------------------------------------- 

- 표는 가능하면 HTML table 구조로 작성한다.
- SVG로 표를 그리는 방식은 이 공정에서 사용하지 않는다.
- 표가 필요한 경우도 도형 생성이 아니라 **content 안의 HTML table**로 처리한다.
- 표는 반드시 **발문 바로 아래**에 배치한다.
- 표를 보기 아래, 해설 아래, 문항 끝으로 내리는 것 금지 

[9-1. 표 위치 규칙]
- 발문 바로 아래 배치
- 발문과 표 사이에 불필요한 설명문 삽입 금지
- 표 다음에 보기 또는 나머지 발문이 이어지는 구조 허용 

[9-2. 표 구조 규칙]
- <table>, <tr>, <td>, <th> 구조 사용
- 행/열 구조가 명확하면 SVG 대신 table 사용
- border, padding, text-align, vertical-align, border-collapse 등은 인라인 스타일 또는 question-table 구조에 맞게 안정적으로 작성 

[9-3. 표 클래스/래퍼 규칙]
- 발문 표는 가능하면 아래 둘 중 하나 사용
  1) <table class="question-table">...</table>
  2) <div class="question-table-wrap"><table>...</table></div> 

[9-4. 표 폭/정렬 규칙]
- 발문 표 기준 max-width 280px
- 왼쪽 정렬 유지
- 가운데 정렬 금지
- 오른쪽 정렬 금지
- 본문보다 과하게 튀는 크기 금지 

[9-5. 표 폰트 규칙]
- 발문 표 기본 font-size: 8.6pt
- line-height: 1.35
- th, td padding: 3px 5px
- 열 수가 많아 특별히 줄여야 할 때만 8pt까지 허용
- 그 이하 금지 

[9-6. 표 내부 수식 규칙]
- 표 내부 수식은 SVG text가 아니라 HTML 셀 내부에서 MathJax가 처리 가능한 형태로 둔다.
- 표 셀 안의 수식도 일반 수식 규칙($...$)을 따른다. 

[9-7. table HTML 금지 규칙]
- <table>, <tr>, <td>, <th> 직전·직후에 <br> 삽입 금지
- 발문 표와 해설용 보조표를 무분별하게 같은 구조로 섞지 않는다.
- .q-box table 범용 제어를 전제로 데이터 작성하지 않는다.
- 발문 전체를 무조건 표 래퍼로 감싸는 방식 금지

--------------------------------------------------
11. 최종 명령
-------------------------------------------------- 

문항을 읽어라.
발문과 보기를 최대한 복원하라.
사소한 OCR 오류는 최소 범위에서만 보정하라.
문제를 직접 풀어라.
난이도 "상"은 강화된 해설 규칙을 적용하라.
해설 원칙을 최우선으로 유지하라.
충돌 시에는 최소 보정 규칙을 적용하라.
보정이 불가하면 answer를 "[정답불가]"로 처리하라.
학생용 해설지에 들어갈 검토 메모, OCR 주석, 참고 문장, 검산 문단은 출력하지 말라.
도형/그래프는 생성하지 말고 필요한 경우 마커만 표기하라.
표준단원은 위 목록 기준으로 정확히 기록하라.
수식과 표는 엔진에서 깨지지 않도록 규칙대로 작성하라.
그리고 그 결과를 code block 안의 window.questionBank 형식 JS 완성본으로만 출력하라.

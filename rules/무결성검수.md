--------------------------------------------------
1. 최상위 원칙
--------------------------------------------------

[1-1. 검수와 수정은 분리]

- 검수 단계에서는 실제 수정하지 않는다.
- 먼저 오류 여부와 근거만 판정한다.
- 수정본, 교체 코드, 전체 JS 재출력은 사용자 승인 전에는 출력하지 않는다.
- 수정은 반드시 사용자 승인 후, 수정 프로토콜에 따라 별도로 수행한다.

[1-2. 원문 성역]

- 발문(content), 보기(choices), 정답(answer), 해설(solution), 메타데이터는 원칙적으로 원문 그대로 존중한다.
- 검수 단계에서 문장을 더 자연스럽게 고치거나 표현을 다듬지 않는다.
- 검수는 “무엇이 문제인지”만 식별한다.
- 원문을 임의로 재작성하지 않는다.
- 정답을 해설에 맞추거나, 해설을 정답에 맞춰 끼워맞추지 않는다.

[1-3. 무단 추정 금지]

- 판독 불가, 구조 불명, 의미 불명 항목은 추정하지 않는다.
- 애매하면 애매하다고 판정한다.
- 보이지 않는 내용을 상상해서 메우지 않는다.
- 실제 파일, 이미지, PNG, SVG, 표, 마스터 테이블을 확인하지 못한 경우 확인하지 못했다고 명시한다.

[1-4. 구조 무결성 우선]

- window.examTitle, window.questionBank, 객체 구조, 배열 구조, 필드명, 따옴표, 쉼표, 괄호, 이스케이프는 검수 최우선 대상이다.
- 구조가 깨지면 내용이 맞아도 실패다.
- JS 문법 오류가 있으면 즉시 FAIL이다.
- id 중복, 배열 누락, 객체 닫힘 오류, 문자열 깨짐은 중대 오류다.

[1-5. 전체 검수 완전성]

- 사용자가 전체 검수를 요청한 경우 처음 문항부터 마지막 문항까지 모두 본다.
- 일부만 보고 전체 통과 판정하지 않는다.
- 후반 문항, 긴 SVG, 긴 해설, 표가 있는 문항도 생략 없이 검수 대상이다.
- “전반적으로 괜찮음” 같은 총평만으로 끝내지 않는다.
- 전체 판정에는 반드시 문항별 근거 또는 검수 근거가 있어야 한다.

[1-6. 최신 공식 기준 우선]

- 메타데이터 검수는 최신 공식 JS아카이브 룰북과 최신 공식 표준단원키 마스터 테이블을 기준으로 한다.
- 동일 이름 또는 유사 이름의 과거 마스터 테이블이 검색되더라도 최종 기준은 프로젝트에 등록된 최신 공식 `# JS아카이브 표준단원키 마스터 테이블.md`로 한다.

--------------------------------------------------
4. 필수 필드 세부 기준
--------------------------------------------------

[4-1. id]

- id는 1부터 시작하여 연속되는 정수여야 한다.
- id 중복은 FAIL이다.
- id 누락은 FAIL이다.
- id 순서가 실제 배열 순서와 크게 어긋나면 WARN 또는 FAIL이다.

[4-2. level]

- level은 문항 난이도를 나타낸다.
- 허용값: "하", "중", "상" (3단계 고정)
- 빈 문자열이면 WARN 또는 보정 필요로 본다.
- "중상", "중하" 등 비허용 값이 있으면 FAIL이다.
- 난이도와 실제 문항 수준이 심하게 맞지 않으면 WARN이다.

[4-3. category / originalCategory]

- category는 현재 분류명이다.
- originalCategory는 원본 또는 세부 분류명이다.
- 둘이 같아도 허용된다.
- 단, 완전히 다른 단원을 가리키면 WARN 또는 FAIL이다.

[4-4. standardCourse / standardUnitKey / standardUnit / standardUnitOrder]

- 네 필드는 반드시 서로 정합해야 한다.
- 최신 공식 마스터 테이블 기준으로 검수한다.
- standardUnitKey와 standardUnit이 불일치하면 FAIL이다.
- standardUnitOrder가 마스터 테이블 순서와 다르면 FAIL 또는 WARN이다.
- standardCourse와 단원키 체계가 맞지 않으면 FAIL이다.

[4-5. questionType]

- questionType은 문항 성격 분류용이다.
- 예: "객관식", "단답형", "서술형"
- questionType만 보고 배치 규칙을 판정하면 안 된다.
- 객관식인데 choices가 없거나, 서술형인데 객관식 보기 구조가 남아 있으면 FAIL 또는 WARN이다.

[4-6. layoutTag]

- layoutTag는 배치 기준이다.
- 기본값은 "grid"로 본다.
- 특수 배치는 layoutTag와 wide 기준으로만 판정한다.
- 누락된 경우 기본값 `"grid"` 보강 대상이며, 누락만으로 즉시 FAIL 처리하지 않는다.

허용 예:
- "grid"
- "subjective-2up"
- "subjective-4up"
- "fullwidth"

[4-7. tags]

- tags는 배열이어야 한다.
- tags는 자동 배치 기준이 아니라 분류/표시용이다.
- tags 누락은 스키마 보강 필요 WARN이다.
- 기본값은 []이다.
- 서술형 문항은 ["서술형"] 권장이다.
- 표 문항은 ["표"] 또는 ["공통자료", "표"] 권장이다.
- 도형 문항은 ["도형"] 권장이다.

[4-8. wide]

- wide는 boolean이어야 한다.
- 기본값은 false이다.
- wide:true는 마스터 명시 지시 또는 fullwidth 문항 전용으로 본다. AI 자동 부여는 금지한다.
- wide 누락은 스키마 보강 필요 WARN이다.
- wide:true만으로 문항 성격을 판단하지 않는다.

[4-9. choices]

- 객관식은 choices가 배열이어야 한다.
- 객관식 보기 수는 원칙적으로 5개다.
- 서술형/단답형은 choices: []를 원칙으로 한다.
- choices 자체가 없으면 구조 결함이다.

[4-10. answer]

- answer는 보기와 실제로 대응해야 한다.
- 객관식은 "①", "②", "③", "④", "⑤" 형태를 원칙으로 한다.
- 서술형은 값 또는 수식으로 명확히 제시되어야 한다.
- 정답이 보기 안에 없으면 FAIL이다.
- 정답 후보가 2개 이상이면 FAIL이다.

[4-11. solution]

- solution은 학생용 해설이어야 한다.
- 빈 문자열은 해설 미작성 상태로 볼 수 있으나, 해설 포함 최종본이라면 WARN 또는 FAIL이다.
- answer와 결론이 일치해야 한다.
- 운영 로그, 생성 흔적, 내부 메모가 있으면 WARN 또는 FAIL이다.

[solution 형식 검수]
- `[키포인트]`로 시작하는가 — 없으면 WARN
- 단계별 풀이가 최소 2단계 이상인가 — 없으면 WARN
- 마지막 줄이 `∴ 정답: ○` 또는 `따라서 정답은 ○이다.` 형태인가 — 없으면 WARN
- cite 태그·내부 메모·생성 로그가 남아 있는가 — 있으면 FAIL
- 최상위 분수가 `\dfrac`으로 작성됐는가 — `\frac` 사용 시 WARN
- 단, 지수/아래첨자 내부 분수(`^{\frac}`, `_{\frac}`)는 `\frac` 허용

--------------------------------------------------
5. 문항 수리 검수
--------------------------------------------------

[5-1. 성립성 검수]

다음을 확인한다.

- 발문이 수학적으로 성립하는지
- 조건이 서로 충돌하지 않는지
- 필요한 값이 누락되지 않았는지
- 정의역, 범위, 자연수/정수/실수 조건이 충분한지
- 해가 존재하는지
- 답이 하나로 결정되는지
- 객관식 보기 중 정답이 존재하는지
- 서술형 정답이 단일하게 확정되는지

[5-2. 계산 검수]

- 각 문항마다 직접 계산하여 정답을 확인한다.
- 해설이 맞아 보인다고 통과시키지 않는다.
- answer 보기만 확인하지 않고, 나머지 보기들이 오답인지도 확인한다.
- 계산 결과와 answer가 다르면 FAIL이다.
- 해설 중간 계산이 틀리면 FAIL 또는 중대 WARN이다.
- 결과는 맞지만 해설 과정이 틀리면 FAIL까지 검토한다.

[5-3. 정답 유일성 검수]

객관식은 반드시 다음을 확인한다.

- 모든 보기를 하나씩 판정한다.
- 정답 후보가 정확히 1개인지 확인한다.
- 정답 후보가 2개 이상이면 즉시 FAIL이다.
- 정답 후보가 0개이면 즉시 FAIL이다.
- answer가 가리키는 보기만 맞는지 확인하고 끝내면 안 된다.
- 발문 표현이 애매하여 정답 후보가 달라질 수 있으면 WARN 또는 FAIL이다.
- “옳은 것”, “옳지 않은 것”, “모두 고른 것”, “아닌 것” 문항은 특히 전 보기 검수를 강제한다.

[5-4. 발문 표현 애매성 검수]

- 수학적으로 엄밀하지 않은 표현이 정답을 바꿀 수 있으면 FAIL 후보로 본다.
- 학생이 다르게 해석할 수 있으나 정답에는 영향이 작으면 WARN이다.
- 예:
  - “제곱근 16은 4이다”는 “√16=4”인지 “16의 제곱근은 4”인지 애매할 수 있다.
  - “완전제곱식”과 “완전제곱식의 상수배”는 구분해야 한다.
  - “공통인수가 아닌 것”은 보기 중 2개 이상이 해당되지 않는지 반드시 확인한다.

- 긴 보기의 번호/본문 정렬이 심하게 무너지면 FAIL이다.

[6-4. 서술형 choices 검수]

  1) 원본 기준과 다른 교육과정 체계로 임의 변경된 경우
  2) 한 문항 내부에서 H15와 H22가 근거 없이 혼합된 경우
  3) standardCourse, standardUnitKey, standardUnit, standardUnitOrder가 서로 불일치하는 경우
  4) 명확한 대응 단원이 있는데 RAW로 방치된 경우

[9-4. RAW 검수]

- RAW 사용 시 standardUnitOrder는 반드시 999인지 확인한다.
- RAW 사용 시 standardUnit은 “미분류” 또는 임시 단원명으로 명확히 구분되어야 한다.

11. SVG / table / PNG / image 필드 세부 검수
--------------------------------------------------

[11-1. SVG 검수]

- SVG 내부 `<br>`는 0개인지 확인한다.
- SVG 내부에 `$...$`, `\frac`, `\sqrt`, `\text` 등 LaTeX 직접 삽입이 있으면 WARN 또는 FAIL이다.
- SVG 내부 `<text>`의 font-size는 원칙적으로 8px, 9px, 10px 범위 기준으로 검수한다.
- 분수 `<g class="fraction">` 내부 숫자 크기는 9px 기준으로 검수한다.
- SVG 좌표, path, text가 비정상적으로 끊기면 FAIL 가능 요소로 본다.
- SVG viewBox가 있는지 확인한다.
- transform 사용이 과도하면 WARN이다.
- 회색 fill은 흑백 인쇄에서 허용 가능하나, 엄격 기준에서는 fill="none" 또는 fill="#fff"가 더 안전하다.
- SVG가 원문 도형 의미와 다르면 FAIL이다.

[11-2. table 검수]

- `<table>`, `<tr>`, `<td>`, `<th>` 닫힘이 맞는지 확인한다.
- `<table>`, `<tr>`, `<td>`, `<th>` 직전·직후에 불필요한 `<br>`가 삽입되어 있으면 WARN 또는 FAIL이다.
- 발문 표에 과도한 인라인 font-size 지정이 있으면 WARN이다.
- 발문 표가 전용 selector 전략 없이 범용 table 제어에 기대고 있으면 WARN 또는 FAIL이다.
- 발문 표는 `.question-table-wrap` 또는 `.question-table` 전용 selector 전략을 기준으로 검수한다.
- `.q-box table` 같은 범용 table selector에 의존하는 구조가 남아 있으면 WARN 또는 FAIL이다.
- 발문 표는 왼쪽 정렬 기준으로 검수한다.
- 발문 표가 가운데 정렬, 오른쪽 정렬, 범용 selector 기반 배치에 기대고 있으면 WARN 또는 FAIL이다.

[11-3. image 필드 검수]

신방식 image 필드를 쓰는 경우 경로 형식은 아래 구조와 맞아야 한다.

- `assets/images/시험지전체명/q{id}.png`

검수 기준:
- image 필드가 있으면 경로 문자열이 올바른지 확인한다.
- 고2 이상 문항은 과목명까지 포함된 시험지전체명 기준 폴더를 사용하는지 확인한다.
- image 필드와 content 내부 `<img>`가 동시에 있어 중복 출력 위험이 있으면 WARN 또는 FAIL이다.
- 신규 데이터는 image 선택 필드를 우선 구조로 보고, content 내부 인라인 `<img>`는 호환 구조로 본다.

[11-4. PNG 실파일 존재 검수]

- 실제 PNG 파일이 함께 제공된 경우: 존재 여부까지 확인하고 없으면 FAIL이다.
- JS만 제공된 경우: 경로 형식만 확인하고 “실파일 존재 여부 미확인 WARN”으로 둔다.
- 로컬/GitHub 실제 파일 목록이 없는데 존재한다고 단정하지 않는다.
- image 필드가 있는데 실제 PNG가 없으면 FAIL 또는 중대 WARN이다.
- PNG가 발문 바로 아래에 배치되지 않으면 WARN 또는 FAIL이다.
- PNG는 불필요한 격자, Export 라벨, 과도한 여백, 축 숫자 과다 여부를 함께 본다.

[11-5. 구방식 img 검수]

- 구방식 `<img src="...">`를 쓰는 경우 실제 프로젝트 구조와 맞는지 확인한다.
- content 내부 `<img>`가 발문 바로 아래에 있는지 확인한다.
- image 필드와 중복되지 않는지 확인한다.
- 경로가 깨졌거나 파일이 없으면 FAIL 또는 중대 WARN이다.

13. 배치 규칙 검수
--------------------------------------------------

[13-1. questionType과 layoutTag 분리]

- 문항 유형(questionType)과 배치 태그(layoutTag)는 분리하여 검수한다.
- questionType은 문항 성격 분류용이며, 직접적으로 페이지 배치 규칙을 결정하지 않는다.
- 배치 판정은 layoutTag, wide 기준으로만 수행되어야 한다.
- tags는 자동 배치 기준이 아니라 분류/표시용으로만 사용되어야 한다.

[13-2. 금지 배치 판정]

다음은 오류로 본다.

- choices.length 기준으로 특수배치를 판정
- choices.length === 0만으로 주관식 특수배치 여부 판정
- questionType만 보고 fullwidth 판정
- tags만 보고 자동 배치 판정
- 주관식 전체를 자동으로 B2/B4 또는 fullwidth로 간주
- wide:true를 자동 추론으로 부여

[13-3. 특수배치 기준]

- `layoutTag === "subjective-2up"`만 B2로 인정한다.
- `layoutTag === "subjective-4up"`만 B4로 인정한다.
- `layoutTag === "fullwidth"` 또는 `wide:true`만 Group C로 인정한다.
- 일반 문항은 layoutTag: "grid", wide: false를 기본으로 본다.

14. 판정 등급
--------------------------------------------------

[14-1. PASS]

아래 조건을 모두 만족하면 PASS다.

- 구조 이상 없음
- 수학적 오류 없음
- 정답 유일성 확보
- 보기-정답 대응 정상
- 해설-정답 일치
- 렌더링 위험 없음
- 메타데이터 정합
- RAW 불필요 잔존 없음
- 도형/표/그래프 삽입 후 마커 잔존 없음
- 배치 규칙 오염 없음
- 보기/서술형 내부 소문항 구조 이상 없음
- JS/PNG 세트 정합 이상 없음
- 즉시 사용 가능

[14-2. WARN]

아래는 WARN에 해당한다.

- 즉시 실행은 가능하나 미세한 위험 또는 보완 필요 요소가 있음
- 일부 수식 표기 불안정
- SVG 내부 표현 경고
- 해설 표현 거칠음
- examTitle/파일명 불일치
- 구조는 살았지만 후속 정리 권장
- 도형/표/그래프 삽입 전 임시 마커가 작업 상태상 아직 남아 있음
- 전용 selector 전략은 있으나 일부 인라인 표기나 폰트 값이 다소 거침
- PNG 파일은 있으나 크기/여백/Export 라벨 정리가 미흡함
- 보기 구조는 동작하나 블록 분리 기준이 다소 불안정함
- image 필드와 구방식 인라인 이미지가 혼재하지만 렌더 자체는 가능한 상태
- JS만 제공되어 image 실파일 존재 여부를 확인할 수 없음
- 공통 자료 문항이 시험지 순서에서는 성립하지만 믹서 독립 문항으로는 불안정함

[14-3. FAIL]

아래는 FAIL에 해당한다.

- JS 구조 오류
- 핵심 필수 필드 누락
- id 중복
- answer-solution 불일치
- 문항 성립 불가
- 정답 복수
- 정답 없음
- 보기-정답 불일치
- 계산 오류
- 렌더링 중대 위험
- 엔진 로드 실패 가능성 큼
- 실제 도형/표/그래프가 삽입되었는데도 기존 마커가 삭제되지 않음
- 배치 규칙이 choices.length 기반 등 금지 방식으로 오염됨
- JS/PNG 세트 누락
- image 필드 경로는 있으나 실제 PNG 자산이 없음
- 문장형 보기 또는 서술형 내부 소문항 구조가 심하게 무너짐
- 명확한 표준단원 대응이 있는데 RAW로 방치됨
- 최신 공식 마스터 테이블과 메타데이터 불일치
- 즉시 수정 필요

18. 최종 행동 명령

고등 메타데이터는 H15와 H22 모두 허용하되, 원본 기준 불일치와 무근거 혼합만 오류로 판정하라.
questionType과 layoutTag를 분리하여 검수하라.
특수배치는 layoutTag와 wide 기준으로만 판정하라.
tags는 분류/표시용으로만 해석하라.
도형 마커나 표 마커나 그래프 마커가 있던 문항은 실제 도형/표/그래프 삽입 후 마커가 삭제되었는지 반드시 확인하라.
문장형 보기와 서술형 내부 `(1)(2)(3)` 소문항 구조를 반드시 확인하라.
PNG와 image 필드는 `assets/images/시험지전체명/q{id}.png` 기준으로 검수하라.
공통 표/공통 그림/공통 자료 문항은 범위 표시와 첫 문항 자료 삽입 여부를 함께 검수하라.
객관식은 모든 보기를 직접 판정하여 정답 유일성을 확인하라.
전체 검수 요청을 받으면 처음부터 끝까지 모든 문항을 빠짐없이 검수하라.
검수 결과는 PASS / WARN / FAIL 중 하나로 판정하고, 근거를 명확히 적시하라.

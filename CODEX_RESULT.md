## 2. 구현 완료 또는 확인 완료

### 학생 등록 중복 방지

- 신입생 등록 중 저장 버튼을 `등록 중...` 상태로 잠그도록 보강했다.
- 저장 중 재클릭 시 추가 POST를 보내지 않고 안내 toast만 표시하도록 했다.
- 신입생 등록 화면에서 수동 PIN 입력칸을 제거했다.
- 학생 생성 API에서 최근 2분 이내 동일 이름/학교/학년/연락처/보호자관계/반 조합의 재원생이 이미 있으면 새로 insert하지 않고 기존 학생 id를 반환하도록 했다.
- 학생 id 생성값에 랜덤 suffix를 추가해 같은 ms 생성 충돌 가능성을 줄였다.
- PIN 자체는 학생 포털/플래너/숙제사진 로그인에 필요하므로 자동 생성 흐름은 유지했다.

### 대시보드/현황 동기화

- 원장 대시보드 `확인 필요`에서 `PIN 미발급` 카드를 제거했다.
- 관리 메뉴의 기본 노출형 `PIN 생성` 진입점을 제거했다.
- 반 정리 필요 집계는 `재원` 학생만 대상으로 보도록 수정했다.
- 숨김/퇴원/휴원 학생이 반 배정 관계를 갖고 있어도 `반 정리 필요`에 잡히지 않도록 정리했다.
- 최근 등록 원생 카드에서 PIN 필요/완료 배지를 제거해 자동 PIN 구조와 맞췄다.
- 학급관리의 학생 수는 재원생 기준으로 계산하도록 보강했다.

### 숨김 학생/중복 생성 정리

- `숨김 학생` 목록을 퇴원생 관리 흐름 안에서 확인할 수 있게 했다.
- 숨김 학생은 복구 또는 완전 삭제를 선택할 수 있게 했다.
- 완전 삭제는 admin 전용이며 `숨김` 상태 학생만 허용한다.
- 출결, 숙제, 시험, 상담, 학교시험, 수납, 결제, 환불, 이월, 교재 오답 등 운영 기록이 있으면 완전 삭제를 막는다.
- 기록이 없는 숨김 학생만 `students`, `class_students`, 보호자 연락처/동의, 메시지 로그, 상태/반 이동 이력, 개인정보 접근 로그 등 연결 row를 정리하고 삭제한다.

### 보존 확인

- 학생 포털/플래너/숙제사진/OMR/QR/아카이브 흐름은 수정하지 않았다.
- Worker schema/migration은 수정하지 않았다.
- 수납·출납 UI는 새로 노출하지 않았다.
- 기존 학생 상세/반 화면/출결/시험 흐름은 유지했다.
- 기존 문구는 요청 범위 안의 PIN/숨김 학생 관리 관련 문구만 조정했다.

## 4. 결과 요약

신입생 등록 중복 클릭으로 같은 학생이 두 번 생성되는 문제를 프론트 버튼 잠금과 백엔드 최근 동일 요청 방어로 함께 막았다. PIN은 자동 생성 기준으로 유지하되 운영 대시보드의 PIN 미발급 확인 항목과 기본 노출형 PIN 생성 진입점은 제거했다. 숨김 학생은 반 정리 필요 집계에서 제외하고, 중복 생성 정리용 완전 삭제는 기록이 없는 숨김 학생에 한해서만 가능하게 했다.

## 5. 잘못한 점 / 위험했던 점 / 보존한 점

- 잘못한 점: 없음
- 위험했던 점: 학생 완전 삭제는 운영 기록 삭제 위험이 크므로 숨김 상태이면서 운영 기록이 없는 학생만 허용하도록 제한했다.
- 위험했던 점: PIN 미발급 카드를 제거하면서 기존 PIN 자동 생성과 학생 로그인 흐름은 반드시 유지해야 했다.
- 위험했던 점: 숨김 학생이 class_students에 남아 있으면 대시보드 반 정리 필요와 반 인원 집계가 어긋날 수 있어 집계 기준을 재원생 중심으로 정리했다.
- 보존한 점: 학생 포털, 플래너, 숙제사진, OMR/QR, 아카이브, schema/migration, 수납·출납 UI 비노출 원칙을 보존했다.
